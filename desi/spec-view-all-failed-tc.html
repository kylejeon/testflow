§4. 상세 기획 — Option B 구현 스펙

4-1. URL 구조

/testcases?status=failed&period=active_run

Query Parameters:
  status   = 'all' | 'passed' | 'failed' | 'blocked' | 'untested' | 'pending' | 'retest'
             (default: 'all' — 전체 TC 표시)
  period   = 'active_run' | '7d' | '30d' | 'all'
             (default: localStorage 'passrate_period' 값 또는 'active_run')
  project  = '{project_id}'  (optional — 특정 프로젝트 필터)
  priority = 'critical' | 'high' | 'medium' | 'low' (optional)
  sort     = 'fail_count' | 'last_failed' | 'priority' | 'title'
             (default: 'last_failed')
  order    = 'asc' | 'desc' (default: 'desc')

4-5. Supabase 쿼리 방향

// Step 1: 기간 필터에 맞는 failed results 전체 조회
const dateFrom = period === '7d' ? 7일전 : period === '30d' ? 30일전 : null;

let query = supabase
  .from('test_results')
  .select('test_case_id, run_id, status, created_at')
  .eq('status', 'failed');

if (period === 'active_run') {
  // active run IDs 먼저 조회
  const { data: activeRuns } = await supabase
    .from('test_runs')
    .select('id')
    .in('status', ['new', 'in_progress', 'paused', 'under_review']);
  query = query.in('run_id', activeRuns.map(r => r.id));
} else if (dateFrom) {
  query = query.gte('created_at', dateFrom);
}
// period === 'all' → 날짜 필터 없음

const { data: failedResults } = await query;

// Step 2: TC별 집계 (failCount, lastFailedAt, lastRunId)
const tcAgg = {};  // group by test_case_id

// Step 3: TC 상세 정보 조회
const { data: tcDetails } = await supabase
  .from('test_cases')
  .select('id, title, priority, assignee, project_id, created_at')
  .in('id', Object.keys(tcAgg));

// Step 4: 프로젝트명 조회
const { data: projects } = await supabase
  .from('projects')
  .select('id, name')

§6. 기존 /testcases 페이지와의 관계

핵심 원칙: 별도 페이지 신설 ✗ → 기존 페이지 강화 ✓

/testcases 페이지는 현재 "모든 TC 탐색" 용도이나, 상태 필터가 미작동 상태입니다.
이번 기획으로 상태 필터를 실제 동작하게 만들면서, URL 파라미터 기반 딥링크까지 지원합니다.
결과: "View All" 뿐 아니라, 앱 어디서든 /testcases?status=XXX 로 이동하여 상태별 TC 리스트를 즉시 확인할 수 있는 범용 기능이 됩니다.

6-1. 상태 필터 탭 UI (신규)

┌─────────────────────────────────────────────────────┐ │ [ All ] [ Failed (47) ] [ Passed ] [ Blocked ] ... │ │ ↑ lifecycle 탭 아래에 execution status 탭 추가 │ │ │ │ 기존: [All] [Draft] [Active] [Deprecated] │ │ → lifecycle_status 필터 (유지) │ │ │ │ 신규: [All] [Failed 47] [Passed 128] [Blocked 5] │ │ → execution status 필터 (추가) │ └─────────────────────────────────────────────────────┘
UI 구조: lifecycle 탭(상단)과 execution status 탭(그 아래)이 독립적으로 동작합니다. 예: "Active" lifecycle + "Failed" status = 활성 TC 중 최근 실패한 것만 표시.

☐Period 필터 변경 시 데이터가 재조회되는가
☐Fail Count 기준 정렬이 정상 동작하는가
☐TC 행 클릭 시 기존 상세 패널이 열리는가
☐"Last Run" 링크 클릭 시 해당 Run Detail로 이동하는가
☐Failed TC가 0건일 때 Empty State가 표시되는가
☐페이지네이션이 20건 단위로 정상 동작하는가
☐딥링크 URL 공유 시 동일 필터 상태가 재현되는가
☐사이드바에서 /testcases 직접 진입 시 status=all (기본값)로 동작하는가
