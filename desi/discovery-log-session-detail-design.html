Discovery Log Sprint 24 Release Payment Flow Edge Cases
Payment Flow Edge Cases
Mission / Charter
Explore edge cases in 3DS verification timeouts, concurrent payment sessions, and partial refund scenarios. Focus on boundary conditions where payment state can become inconsistent between our system and the payment gateway.
10:30 AM — Session Started

3DS Verification Timeout

Testing 3DS verification flow with artificial 30-second timeout. The payment gateway returns a timeout error but our system doesn't handle it gracefully.

3DS timeout after 30s leaves payment in "processing" state — no retry or cancel option shown to user critical

Steps to reproduce: Navigate to checkout → select "3DS Required" card → wait 30 seconds → observe the spinner stays indefinitely.

The expected behavior per Stripe docs is to receive a payment_intent.requires_action webhook, but we're not listening for it.
10:52 AM

Concurrent Payment Sessions

Opened two browser tabs, attempted to pay for the same order simultaneously. Both succeeded — double charge on the customer's card.

No idempotency check — concurrent payment creates duplicate charges critical
  • Test 3DS timeout (30s, 60s, 90s)
  • Test concurrent sessions (2 tabs)
  • Test partial refund edge cases
  • Test network interruption during 3DS
11:15 AM

Partial Refund Rounding

When processing a partial refund of $33.33 on a $99.99 order, the remaining balance shows as $66.65 instead of $66.66. This is a classic floating-point rounding issue.

Currency rounding error — $0.01 discrepancy in partial refund calculations major
