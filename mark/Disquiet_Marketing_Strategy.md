## 1. 디스콰이엇이란?

디스콰이엇(disquiet.io)은 한국 인디 메이커들이 사이드 프로젝트와 스타트업을 소개하고, 빌딩 과정을 공유하며, 서로 피드백을 주고받는 커뮤니티입니다.

Product Hunt의 한국판이라고 흔히 불리지만, 결이 조금 다릅니다. Product Hunt가 "런칭 이벤트"에 집중한다면, 디스콰이엇은 **"만들어가는 과정"** 자체를 콘텐츠로 봅니다. 완성된 제품보다 만드는 사람의 스토리와 여정이 더 주목받습니다.

**주요 기능:**

- **메이커로그:** 일간/주간 빌딩 업데이트. "오늘 이걸 만들었어요" 형식의 짧은 기록
- **제품 페이지:** 제품 소개, 스크린샷, 링크, 메이커 정보 등록
- **스토리:** 긴 글 형식의 메이커 여정 공유
- **좋아요 / 응원:** 커뮤니티 반응
- **팔로우:** 메이커 구독
- **댓글 / 피드백:** 직접적인 의견 교환
- **주간 뉴스레터:** 이번 주 주목할 메이커/제품 큐레이션

## 2. 디스콰이엇 오디언스와 문화

**주요 사용자:**
- 사이드 프로젝트를 운영하는 개발자 / 디자이너
- 초기 스타트업 창업자
- 인디 해커 / 1인 창업자
- 제품에 관심 있는 PM, 마케터

**핵심 문화:**
- **솔직함이 미덕:** 실패담, 느린 성장, 막히는 부분을 솔직하게 공유할수록 반응이 좋습니다
- **숫자 공유를 좋아함:** "이번 주 사용자 3명 늘었어요", "MRR $0 → $47" 같은 작은 숫자도 환영
- **사이드 프로젝트를 존중:** 본업이 따로 있으면서 만드는 것을 "열정"으로 봅니다
- **커뮤니티 중심:** 조회수보다 댓글 대화를 더 중요하게 여깁니다
- **마케팅 냄새를 싫어함:** 광고처럼 느껴지는 포스트는 즉시 묻힙니다

## 3. LinkedIn vs 디스콰이엇 — 전략 차이

| 구분 | LinkedIn (Kyle) | 디스콰이엇 |
|------|----------------|-----------|
| 타겟 | QA 리드, Engineering Manager | 인디 메이커, 스타트업 창업자 |
| 역할 | 전문가 브랜딩 / 잠재 고객 확보 | 메이커 커뮤니티 / 피드백 수집 |
| 톤 | 전문가 인사이트, 차분한 소개 | 솔직한 여정 공유, 취약함 OK |
| 콘텐츠 | QA 업계 인사이트, 제품 소개 | 빌딩 과정, 배운 것, 실패담 |
| Testably 언급 | Phase 1은 자제, 이후 자연스럽게 | 처음부터 제품이 핵심 소재 |

| 언어 | 한글 2회 + 영문 2회 | 한글 우선 |

**핵심 차이:** LinkedIn에서 Kyle은 "QA 전문가"입니다. 디스콰이엇에서 Kyle은 "뭔가를 만들고 있는 사람"입니다.

## 4. 포지셔닝 전략

**디스콰이엇에서의 Testably 포지션:**

> "QA 팀들이 매일 쓰는 도구가 왜 이렇게 불편할까? 라는 질문으로 시작한 사이드 프로젝트"

이게 핵심 메시지입니다. 거창한 비전이나 시장 규모 이야기 대신, "나 이거 불편해서 직접 만들었어요"가 디스콰이엇에서 가장 공감을 얻는 스토리입니다.

**강조할 것:**
- 직장인이 새벽/주말 시간을 모아 만들었다는 사실
- 거창한 목표보다 "일단 내가 쓰고 싶은 도구"에서 출발했다는 점
- 현재 단계(베타, 초기)를 숨기지 않고 오히려 강조
- 만들면서 겪는 어려움, 막히는 부분도 솔직하게 공유

**피해야 할 것:**
- "혁신적인", "게임체인저" 같은 과장된 표현
- 경쟁사 디스
- 완성된 척하는 포스팅
- 조회수/사용자 수 자랑 (초기엔 특히)

### 제품 소개 (본문)
```
Testably는 QA 팀을 위한 테스트 케이스 관리 도구입니다.

만든 이유는 단순합니다. QA 팀들이 테스트 케이스를 관리하는 방식을 오랫동안 지켜봤는데, 대부분이 스프레드시트를 쓰거나, 쓰기엔 너무 무거운 엔터프라이즈 도구를 억지로 사용하고 있었습니다. 그 중간 어딘가에 "이 정도면 딱 좋겠다"는 도구가 없었습니다.

그래서 만들었습니다. 본업을 하면서 새벽과 주말 시간에 조금씩.

Testably가 지향하는 것:
- 5분 안에 첫 테스트 케이스를 만들 수 있는 단순함
- 테스트 케이스 ↔ 실행 결과 ↔ 버그가 자연스럽게 연결되는 구조
- 팀 누구나 배우지 않아도 바로 쓸 수 있는 UI
- 스프레드시트에서 쉽게 마이그레이션

아직 완성된 제품이 아닙니다. 지금은 같은 문제를 느끼는 분들과 함께 만들어가는 단계입니다. 피드백 환영합니다.
```

### 카테고리 (해당 항목 선택)
```
개발 도구 / 생산성 / B2B SaaS
```

### 현재 상태
```
베타 / 얼리 액세스
```

### 가격 정책
```
얼리 액세스 기간 무료 (추후 유료 플랜 예정)
```
*(실제 가격 정책에 맞게 수정해주세요)*

### 메이커 한마디 (개인 소개란)
```
본업을 하면서 사이드 프로젝트로 Testably를 만들고 있습니다. QA와 소프트웨어 품질에 오랫동안 관심을 가져왔고, 직접 쓰고 싶은 도구가 없어서 만들기 시작했습니다. 매주 여기에 만들어가는 과정을 공유할 예정입니다.
```

## 6. 디스콰이엇 알고리즘 & 노출 메커니즘

**노출에 영향을 주는 것:**

1. **초기 반응 속도:** 포스팅 후 첫 1~2시간 내 좋아요/댓글이 많을수록 상단 노출 유지
2. **꾸준한 메이커로그:** 매주 꾸준히 업데이트하는 메이커가 알림 구독자에게 노출
3. **댓글 참여:** 내 포스트에 댓글이 달리면 알고리즘이 활성 콘텐츠로 인식
4. **타 메이커 포스트에 댓글:** 상대방의 팔로워에게 Kyle의 이름이 노출
5. **주간 뉴스레터 픽업:** 운영팀이 흥미로운 메이커/제품을 큐레이션. 솔직하고 진정성 있는 포스트가 선택되는 경향
6. **팔로워 알림:** 팔로워가 생기면 새 포스트 발행 시 알림이 감

**노출을 방해하는 것:**
- 너무 자주 올리기 (하루 여러 개는 역효과)
- 광고/홍보처럼 느껴지는 톤
- 댓글 없이 일방적 발행만 하기
- 다른 메이커 포스트에 무반응

## 9. 디스콰이엇 베스트 프랙티스

**해야 할 것:**

- **꾸준함이 전부:** 한 달에 2개 올리는 것보다 매주 1개씩 꾸준히 올리는 게 훨씬 효과적. 구독자가 생기면 업데이트를 기다립니다
- **댓글에 반드시 답변:** 댓글이 달리면 무조건 답하세요. 대화가 길어질수록 노출이 늘어납니다
- **다른 메이커 응원하기:** 비슷한 단계의 메이커들 포스트에 진심 어린 댓글을 남기면 자연스럽게 관계가 생깁니다
- **숫자 공유:** 작아도 됩니다. "이번 주 가입자 2명"도 콘텐츠입니다. 투명함이 신뢰를 만듭니다
- **메이커로그 활용:** 스토리 형식보다 짧은 메이커로그를 자주 올리는 게 알고리즘에 유리합니다
- **이미지 첨부:** 스크린샷, 그래프, 작업 화면 등 시각 자료가 있으면 클릭률이 높아집니다

**피해야 할 것:**

- **같은 날 여러 개 포스팅:** 스팸처럼 보이고 커뮤니티 반응이 차가워집니다
- **"써보세요!" CTA만 있는 포스트:** 스토리 없이 홍보만 하는 포스트는 묻힙니다
- **완벽한 척:** 오히려 역효과. 솔직하게 "이 부분은 아직 미흡해요"가 더 반응이 좋습니다
- **다른 메이커 무시:** 내 포스트만 올리고 커뮤니티 활동을 안 하면 팔로워가 늘지 않습니다
- **LinkedIn 포스트 그대로 복붙:** 디스콰이엇 톤과 맞지 않아 어색하게 느껴집니다
