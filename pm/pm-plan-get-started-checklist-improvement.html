Executive Summary

CEO 피드백: "워크플로우가 명확하지 않고, 클릭해도 동작하지 않는 항목이 있다."

소스코드 분석 결과, 6개 항목 중 자동 완료가 되는 것은 1개(Create Project)뿐이며, 나머지 5개는 markStep 호출이 없어 사용자가 실제 액션을 수행해도 체크되지 않습니다. 또한 네비게이션 링크에 버그가 있어 Settings 탭 이동이 작동하지 않습니다.

추가로, 두 개의 체크리스트 컴포넌트가 공존하며 (OnboardingChecklist + GettingStartedChecklist) 후자는 사용되지 않는 데드코드입니다.

2. 핵심 발견 — 버그 및 미구현 항목

🔴 Critical Bug 1: 자동 완료 감지 미구현 (5/6 항목)
markStep() 호출이 코드베이스 전체에서 createProject 한 곳에서만 발생합니다. 나머지 5개 스텝(createTestcase, tryAi, runTest, inviteMember, connectJira)은 사용자가 실제로 수행해도 DB에 반영되지 않아 체크가 되지 않습니다.
🔴 Critical Bug 2: Settings 탭 네비게이션 불일치
OnboardingChecklist가 /settings#team, /settings#integrations로 링크하지만, Settings 페이지는 해시(#) 대신 쿼리 파라미터(?tab=members)를 사용합니다. 결과적으로 클릭 시 Settings 페이지로 이동은 되지만 Profile 탭(기본값)이 표시됩니다.
🟡 Issue 3: 데드코드 — GettingStartedChecklist.tsx
pages/projects/components/GettingStartedChecklist.tsx는 어디에서도 import되지 않는 미사용 파일입니다. localStorage 기반의 다른 체크리스트인데, OnboardingChecklist (DB 기반)와 혼동됩니다. 삭제해야 합니다.
🟡 Issue 4: 항목 간 의존성 없음
"프로젝트 생성 전에 TC 작성 불가"와 같은 논리적 순서가 있으나, 모든 항목이 독립적으로 클릭 가능합니다. 프로젝트가 없는 상태에서 TC 작성 클릭 → /projects 페이지로 이동하지만 맥락이 없습니다.
🟡 Issue 5: 수동 토글 vs 자동 감지 혼합 설계
GettingStartedChecklist (데드코드)는 수동 클릭으로 체크/언체크가 되고, OnboardingChecklist (활성)는 DB 기반이지만 자동 감지가 없어 사실상 아무것도 체크되지 않는 상태입니다.

4. 개선안 — 워크플로우 재설계

4.1 핵심 원칙

🎯 클릭 = 즉시 맥락

각 항목 클릭 시 해당 기능의 정확한 위치로 이동하고, 가능하면 관련 모달/UI를 자동으로 열어줍니다.

✅ 완료 = 자동 감지

사용자가 수동으로 체크하는 것이 아니라, 실제 액션 완료 시 DB에 자동 반영됩니다.

🔗 순서 = 의존성 안내

선행 스텝 미완료 시 비활성화하지는 않되, 인라인 안내를 제공합니다.

🧹 정리 = 데드코드 삭제

GettingStartedChecklist.tsx를 삭제하고, 단일 시스템으로 통합합니다.

5.2 의존성 처리 방식 — Soft Lock 권장

권장 방식: Soft Lock (비활성화하지 않되 안내 제공)
모든 항목은 항상 클릭 가능하지만, 선행 조건 미충족 시 이동 후 인라인 토스트로 안내합니다. Hard Lock(비활성화)은 사용자 자유도를 제한하므로 비권장합니다.

안내 메시지 예시:

  • Step 2/3 클릭 + 프로젝트 없음 → "프로젝트를 먼저 만들어야 테스트 케이스를 추가할 수 있어요."
  • Step 4 클릭 + TC 없음 → "테스트 케이스를 먼저 추가해야 테스트 실행을 만들 수 있어요."

// useOnboarding.ts — fetchOnboarding 내부에 추가
async function reconcileSteps(userId: string) {
  // 프로젝트 존재 확인
  const { count: projectCount } = await supabase
    .from('project_members').select('*', { count: 'exact', head: true })
    .eq('user_id', userId);

  // TC 존재 확인
  const { count: tcCount } = await supabase
    .from('test_cases').select('*', { count: 'exact', head: true })
    .eq('created_by', userId);

  // Run 존재 확인
  const { count: runCount } = await supabase
    .from('test_runs').select('*', { count: 'exact', head: true })
    .eq('created_by', userId);

  // Jira 연결 확인
  const { data: jira } = await supabase
    .from('jira_settings').select('id').eq('user_id', userId).single();

  const updates: Record<string, boolean> = {};
  if (projectCount > 0) updates.step_create_project = true;
  if (tcCount > 0) updates.step_create_testcase = true;
  if (runCount > 0) updates.step_run_test = true;
  if (jira) updates.step_connect_jira = true;
  // tryAi, inviteMember는 프론트엔드에서만 감지 (별도 테이블 없음)

  if (Object.keys(updates).length > 0) {
    await supabase.from('user_onboarding').update(updates).eq('user_id', userId);
  }
}

App.tsx에서 OnboardingProvider로 감싸고, 하위 컴포넌트에서 useOnboardingContext()로 markStep에 접근합니다. state와 markStep을 단일 소스에서 관리하여 일관성을 보장합니다. 
// contexts/OnboardingContext.tsx
const OnboardingContext = createContext<UseOnboardingReturn | null>(null);

export function OnboardingProvider({ children }: { children: React.ReactNode }) {
  const onboarding = useOnboarding();
  return (
    <OnboardingContext.Provider value={onboarding}>
      {children}
    </OnboardingContext.Provider>
  );
}

export function useOnboardingContext() {
  const ctx = useContext(OnboardingContext);
  if (!ctx) throw new Error('OnboardingProvider not found');
  return ctx;
}
