1. 개요 및 현재 상태 분석

배경

Dashboard의 SparseState(프로젝트 1개)에서 표시되는 TipsBanner에는 사용자의 다음 단계를 유도하는 버튼들이 있습니다. 현재 이 버튼들은 onClick 핸들러가 없거나 빈 함수로 되어 있어 클릭해도 아무 동작을 하지 않습니다. 이 기획서는 각 버튼의 클릭 시 동작을 정의합니다.

TipsBanner 현재 상태

위치 컴포넌트 버튼 텍스트 현재 onClick 구현 상태
SparseState (count=1) TipsBanner Create Test Case 없음 (미구현) 신규 구현 필요
SparseState (count=1) TipsBanner Explore Sample 없음 (미구현) 기존 로직 연결
EmptyState / SparseState ActionCard Import from TestRail () => {} (빈 함수) 신규 구현 필요
참고: TipsBanner 표시 조건
프로젝트 수에 따라 TipsBanner 내용이 변경됩니다.
count=1: "Create Test Case" (primary) + "Explore Sample" (secondary)
count=2: "Invite Team" (primary) + "Create Milestone" (secondary)
이 기획서는 count=1 상태의 버튼과, ActionCard의 "Import from TestRail"을 다룹니다.

관련 기존 구현체

기능 파일 비고
Quick Create TC Modal QuickCreateTCModal.tsx Title, Folder, Priority, Type 필드. 기존에 작동 중
Sample Project 생성 useSampleProject.ts "Sample E-commerce App" 생성. ActionCard에서 이미 사용 중
TestRail CSV Import testRailExport.ts CSV 파일 기반 파싱만 존재. API 연동 없음
Export/Import Modal ExportImportModal.tsx CSV 파일 업로드 → 미리보기 → 임포트

2.3 엣지 케이스

케이스 조건 처리
Edge 프로젝트 0개 프로젝트가 없는 상태에서 버튼이 노출되는 경우 (이론적으로 불가 — SparseState는 1개 이상) 방어 코드: 프로젝트 목록이 비어있으면 "Create Project" 모달을 대신 열고, 프로젝트 생성 후 TC 생성 모달로 연결
Edge Title 빈 값 제출 Title 없이 Create 클릭 Title 필드 아래에 인라인 에러 표시: "Title is required". Create 버튼 비활성화 상태 유지
Edge API 에러 생성 API 호출 실패 에러 토스트 표시: "Failed to create test case. Please try again.". 모달은 닫지 않고 유지하여 재시도 가능
Edge 중복 제출 Create 버튼 연속 클릭 Create 버튼에 로딩 스피너 표시 + 비활성화. API 호출 중 중복 요청 방지
Edge 모달 외부 클릭 / ESC 실수로 닫기 입력값이 있으면 확인 다이얼로그: "Discard changes?". 빈 상태면 바로 닫기

영역 보이는 내용 사용자 액션 유도

케이스 조건 처리

4.1 현재 구현 vs 신규 기획

항목 현재 구현 (CSV 파일 기반) 신규 기획 (API 연동)
인증 방식 없음 (파일 업로드) TestRail API Key + Instance URL
데이터 소스 사용자가 내보낸 CSV 파일 TestRail API 직접 호출
임포트 데이터 Test Cases만 Projects, Suites, Cases, Runs, Results
매핑 CSV 컬럼 자동 매핑 (flexible column name matching) TestRail 필드 → Testably 필드 매핑 UI
프로세스 파일 선택 → 파싱 → 미리보기 → 임포트 인증 → 데이터 선택 → 매핑 → 진행 → 완료
중요: 두 가지 임포트 경로 제공
기존 CSV 임포트는 유지하되, 새로운 API 연동 임포트를 권장 방식(recommended)으로 제공합니다. 사용자에게 선택지를 주되, API 연동을 primary로 노출합니다.

권장 방식: 서버에서 import job을 생성하고, 프론트엔드가 GET /api/import-jobs/{jobId}/status를 2초 간격으로 폴링합니다.
배치 처리: 대량 데이터의 경우 100건 단위로 배치 삽입하여 DB 부하를 관리합니다.
재시도: 개별 API 호출 실패 시 최대 3회 재시도, 그래도 실패하면 해당 항목은 스킵하고 로그에 기록합니다.

TestRail API Key는 프론트엔드에서 직접 사용하지 않고, 반드시 백엔드 프록시를 통해 호출합니다. API Key는 import 세션 동안만 서버 메모리에 유지하고, 영구 저장하지 않습니다. 사용자에게도 "Your API key is never stored permanently" 메시지로 안심시킵니다.
