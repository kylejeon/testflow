§1. PM 최종 견해 (Executive Summary)

✅ 2-버튼 구조는 유지하되, 3가지 핵심 개선이 필요합니다

현재 "Pass & Next" + "Add Result" 2-버튼 구조는 업계 표준(TestRail·Qase·Zephyr 모두 유사)이며, "빠른 실행"과 "상세 기록"의 명확한 분리는 올바른 설계입니다.

그러나 코드 분석 결과 사용성 문제 6건과 기능 불완전 3건을 발견했습니다. 핵심 개선 3가지:

  1. "Pass & Next"를 인라인 Quick Status 버튼 바로 확장 — Fail & Next, Block & Next 추가
  2. Add Result 모달의 조건부 UI 간소화 — Status에 따라 필드 표시/숨김
  3. 리치 텍스트 에디터 도구 바 작동화 + 노트 저장 버그 수정

2-3. 장단점 분석

✅ 현재 구조의 장점

#장점설명
1 ✓ Happy Path 최적화 대부분의 TC가 Pass되는 일반적 테스트 시나리오에서 1-클릭으로 빠르게 진행 가능. 테스트 실행 효율 극대화.
2 ✓ 명확한 역할 분리 "빠른 Pass"와 "상세 기록"이 물리적으로 분리되어 혼동 없음.
3 ✓ resolved_at 자동 처리 Pass & Next에서 이전 상태가 failed/blocked였으면 resolved_at 자동 설정 (Line 604-614). Avg Response Time 측정에 기여.
4 ✓ 업계 표준 패턴 TestRail(Quick Status + Full Edit), Qase(Wizard + Detail), Zephyr(Test Player + Edit) 모두 유사한 2-tier 구조.

❌ 현재 구조의 단점

#단점심각도설명
1 ✗ Fail 빠른 처리 불가 High TC 실패 시 반드시 "Add Result" 모달을 열어야 함. 단순 실패도 3+ 클릭 필요. Failed가 많은 Regression 테스트에서 비효율.
2 ✗ 모달 후 자동 진행 없음 Medium Add Result 모달에서 결과 제출 후 같은 TC에 머무름. 테스터가 수동으로 다음 TC로 이동해야 함.
3 ✗ 모달 필드 과다 노출 Medium Pass 선택 시에도 Issues, Elapsed Time 등 불필요한 필드가 모두 표시됨. 인지 부담 증가.
4 ✗ 리치 텍스트 비작동 High 노트의 리치 텍스트 도구 바(Bold, Italic 등) 버튼이 UI만 존재하고 실제 기능이 연결되지 않음.
5 ✗ Focus Mode 노트 미저장 High Focus Mode에서 입력한 노트가 _note로 무시됨 (이전 분석에서 발견). 테스터의 기록이 유실.
6 ✗ 모달 ↔ Focus Mode 불일치 Medium Focus Mode에는 Issues/Attachments/Step결과가 없고, 모달에는 키보드 단축키가 없음. 두 모드 간 기능 격차가 큼.

4-2. 모달 UX 문제점 상세

🔴 문제 1: 리치 텍스트 도구 바 비작동 (High)
Bold, Italic, Underline, Strikethrough, Code, Link, List 버튼이 모두 UI만 존재하고 실제 기능이 없습니다. 버튼에 onClick 핸들러가 없으며, textarea는 plain text만 지원합니다. 사용자가 서식을 적용하려고 클릭해도 아무 반응 없음 → 신뢰도 하락.
🔴 문제 2: 모든 상태에 동일한 필드 노출 (Medium)
"Passed" 선택 시에도 Issues(Jira 연동), Elapsed Time, Step별 결과 등이 모두 표시됩니다. Pass 시에는 대부분 노트와 첨부만 필요합니다. 7개 섹션이 한 번에 표시되면 스크롤 양이 많아지고, "무엇을 채워야 하는지" 혼란이 생깁니다.
🟡 문제 3: 결과 제출 후 자동 진행 없음 (Medium)
"Add result" 클릭 후 모달만 닫히고 같은 TC에 머무릅니다. 연속 테스트 시 매번 수동으로 "→" 버튼을 눌러야 합니다. 경쟁사(Qase)는 "제출 후 다음 미완료 TC로 자동 이동" 옵션을 제공합니다.
🟡 문제 4: Step 결과 기록의 시각적 불명확 (Low)
Step별 결과 드롭다운이 작은 select 요소로 되어 있어 어떤 step이 fail인지 한눈에 보기 어렵습니다. TestRail은 step별로 색상 코드 뱃지를 제공하여 시각적 구분이 명확합니다.
🟡 문제 5: Issues 입력의 낮은 직관성 (Low)
"Enter Jira issue key" 입력 필드는 Enter/Comma로 추가하는 방식인데, 이 인터랙션이 시각적으로 안내되지 않습니다. 또한 기존 Jira 이슈 검색/자동완성이 없어 사용자가 이슈 키를 정확히 기억하고 있어야 합니다.

§6. "Pass & Next" 역할 재정의 — Fail & Next, Block & Next 필요한가?

✅ PM 권장: Yes — Quick Status 버튼 바로 확장 (개선안 A)

근거:

  1. 실제 사용 데이터 기반 추론: Regression 테스트에서 Fail/Block 비율이 10-30%일 수 있음. 이 경우 매번 모달을 열면 테스트 실행 시간이 크게 증가.
  2. Focus Mode와의 일관성: Focus Mode는 이미 P/F/B/R/S 5개 버튼을 모두 제공. Run Detail 패널에서만 Pass만 빠른 처리가 가능한 것은 일관성 부족.
  3. 경쟁사 표준: TestRail, Qase, Zephyr 모두 어떤 상태든 1-클릭으로 빠르게 변경 가능.
  4. Fail 시 노트 입력은 인라인 팝업으로 해결: 풀 모달이 아닌 작은 팝업으로 "빠르게 메모 + 다음"을 지원하되, 상세 기록이 필요하면 "Open full modal"로 전환.

단, 모든 5개 상태를 동일 크기 버튼으로 나열하면 시각적 혼잡합니다. Primary/Secondary 구분을 권장합니다:

구분버튼스타일근거
Primary Pass Fail 컬러 배경, 큰 크기 가장 빈번한 2가지 결과
Secondary Block Retest 아웃라인 스타일, 약간 작은 크기 상대적으로 덜 빈번
Tertiary Skip (Untested) "→" 버튼에 통합 또는 별도 텍스트 링크 드물게 사용

§7. 3개 실행 모드 통합 전략

현재 Testably에는 테스트 실행을 위한 3개의 독립적 인터페이스가 존재합니다:

모드현재 상태강점약점적합 시나리오
① Run Detail 패널
(Quick Status + Add Result)		 메인 실행 환경 TC 리스트 + 상세 + 결과를 한 화면에서 Pass만 빠른 처리, 모달 전환 비용 개별 TC 상세 확인 + 결과 입력
② Focus Mode
(풀스크린 키보드 중심)		 미니멀 실행 환경 키보드 단축키, 집중 UX 첨부/Issues/Step결과 없음, 노트 미저장 대량 TC 빠른 실행
③ Add Result 모달
(풀 기능 입력 폼)		 상세 기록 환경 모든 필드 지원 (7개 섹션) 자동 진행 없음, 필드 과다 노출 Failed/Blocked 상세 기록
장기 비전: 3개 모드가 각각의 역할을 유지하되, 기능 격차를 줄이고 전환을 부드럽게 하는 것이 목표입니다.
• Focus Mode → "Open Full Details" 버튼으로 Add Result 모달 진입 가능
• Add Result 모달 → "Submit & Next" 옵션으로 자동 진행
• Quick Status 바 → Fail/Block 시 인라인 팝업 → "Open full modal" 링크
결과: 어떤 모드에서 시작하든 필요에 따라 상세 레벨을 올리거나 내릴 수 있는 "Escalation Path"가 완성됩니다.

§10. CEO 질문에 대한 직접 답변

CEO 질문PM 답변
"이 워크플로우를 유지하는 게 좋을까?" 2-버튼 구조(Quick vs Detail)는 유지합니다. 업계 표준이며 올바른 설계입니다. 다만 Quick 버튼을 Pass 전용에서 모든 상태 지원으로 확장해야 합니다.
"사용성 개선이 필요할까?" Yes, 3가지 핵심 개선이 필요합니다:
① Quick Status 바 확장 (Fail/Block도 1-클릭)
② 모달 조건부 UI (Status에 따라 필드 접힘)
③ 비작동 리치 텍스트 도구 바 정리 + 모달 "Add & Next" 추가
"Fail & Next, Block & Next도 필요한지?" Yes. 경쟁사 모두 지원하며, Focus Mode도 이미 5개 상태를 모두 Quick으로 지원합니다. Run Detail에서만 Pass만 가능한 것은 불일치이자 비효율입니다. 단, Fail/Block은 인라인 노트 팝업을 거치도록 합니다.
