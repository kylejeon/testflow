1 CEO 피드백 — 문제 정의

문제 1: 리치 텍스트 에디터의 사용성

"하나의 rich text box에서 텍스트 형태로 계속 작성하는 방식이면, 모든 사용자가 디자인 가이드처럼 마크업을 예쁘게 사용하면서 작성하지 않을 것 같다."

핵심 문제: 자유 형식 리치 텍스트 에디터는 (1) 사용자마다 작성 품질 차이가 크고, (2) 구조화된 데이터 추출이 어렵고, (3) 나중에 세션 리포트 자동 생성 시 파싱이 불가능합니다. 탐색적 테스팅은 빠른 기록이 핵심인데, 서식 도구바를 신경 써야 하면 기록 속도가 느려집니다.

문제 2: Activity 히트맵 기준 불명확

"히트맵이 어떤 기준으로 표시되는 거야?"

핵심 문제: 기존 기획(§9-7)에서 "5분 단위, 5단계 컬러"만 정의했을 뿐, 측정 대상, 수집 방법, 실제 구현 가능성에 대한 구체적 정의가 없습니다.

2 Session Log 입력 방식 — 옵션 비교 분석

옵션 A: 순수 리치 텍스트 에디터 (현재 기획안)

자유 형식 에디터 — Google Docs / Notion 스타일

하나의 큰 에디터 영역에서 테스터가 자유롭게 작성. H1/H2, Bold, 이미지, 테이블 등 리치 서식 지원.

옵션 A 와이어프레임
┌──────────────────────────────────────────────┐
│ [H1][B][I][U][S][Code][Img][Table][Link] │
│────────────────────────────────────────────── │
│ Found that 3DS timeout after 30s causes the │
│ payment to fail silently. No error message. │
│ │
│ Tried concurrent sessions — both succeeded │
│ but one showed wrong currency symbol. │
│ [screenshot.png] │
│ │
│ (자유 텍스트, 구조 없음) │
└──────────────────────────────────────────────┘

✅ 장점

  • 최대 자유도 — 어떤 형식이든 작성 가능
  • 학습 비용 제로 (모두가 아는 워드 에디터)
  • 구현 비용 낮음 (TipTap/ProseMirror 바로 적용)

❌ 단점

  • CEO 지적 — 작성 품질 편차 큼
  • 구조화 데이터 추출 불가 (리포트 자동 생성 어려움)
  • 버그 vs 관찰 vs 메모 구분 안 됨
  • 나중에 "이 세션에서 버그가 몇 개?" 집계 불가

경쟁사 사례: TestRail의 기존 세션 노트 방식. 단순하지만 데이터 활용도가 낮아 2025년 AI 분석 기능을 추가함.

옵션 B: 구조화된 로그 엔트리 (Structured Log)

시간순 로그 엔트리 — Slack/채팅 스타일

각 엔트리가 독립된 카드/블록으로 생성됩니다. 엔트리 생성 시 유형 선택 필수: Note, Bug, Observation, Step, Screenshot. 각 엔트리에 타임스탬프가 자동 부여됩니다.

옵션 B 와이어프레임
┌──────────────────────────────────────────────┐
│ 10:32 AM · [Note]                       │
│ Starting payment flow exploration. Focus on │
│ 3DS verification edge cases. │
│───────────────────────────────────────────── │
│ 10:45 AM · [Bug] 🐛 Severity: Major     │
│ 3DS timeout after 30s causes silent failure. │
│ No error message shown to user. │
│ 📷 [screenshot-001.png] │
│───────────────────────────────────────────── │
│ 10:52 AM · [Observation]            │
│ Concurrent payment sessions both succeed but │
│ one shows wrong currency symbol (€ vs $). │
│───────────────────────────────────────────── │
│ 11:03 AM · [Step]                      │
│ Tested: Add item → Checkout → 3DS popup │
│ Expected: Complete within 60s │
│ Actual: Timed out at 30s, no retry option │
├──────────────────────────────────────────────┤
│ [+Note] [+Bug] [+Observation] [+Step] [+📷] │
└──────────────────────────────────────────────┘
엔트리 유형아이콘컬러필수 필드선택 필드
Noteri-sticky-note-line#6366F1내용 텍스트—
Bugri-bug-line#EF4444제목 + 내용심각도 (Critical/Major/Minor/Trivial) + 스크린샷 + 외부 이슈 링크
Observationri-eye-line#F59E0B내용 텍스트관련 영역 태그
Stepri-footprint-line#22C55ETested (행위) + Actual (결과)Expected (기대 결과)
Screenshotri-screenshot-line#3B82F6이미지캡션 텍스트 + 어노테이션

✅ 장점

  • 구조화된 데이터: 엔트리 유형별 자동 집계 가능 (버그 3개, 관찰 5개 등)
  • 작성 품질 일정 — 유형별 최소 필수 필드 보장
  • 타임스탬프 자동 → 시간순 타임라인 자연 생성
  • 리포트 자동 생성 쉬움 (유형별 필터링 + 요약)
  • AI 분석 데이터 품질 우수 (구조화 → 분석 용이)

❌ 단점

  • 유형 선택이 기록 속도를 늦출 수 있음
  • 유형 구분이 애매한 경우 사용자 혼란 (Bug vs Observation 경계)
  • 긴 서술형 내용 작성 시 불편 (각 엔트리 텍스트 영역이 작음)
  • 기존 리치 텍스트 대비 자유도 감소

경쟁사 사례: Qase의 탐색적 세션은 구조화된 로그 방식을 채택. Session-Based Test Management(SBTM) 업계 표준과 부합.

옵션 C: 하이브리드 — 구조화 로그 + 리치 텍스트 (Recommended)

구조화 엔트리 기본 + 각 엔트리 내 리치 텍스트

옵션 B의 구조화된 로그 엔트리를 기본으로 하되, 각 엔트리의 내용 필드에서 리치 텍스트(Bold, 이미지, 코드블록, 링크 등)를 사용할 수 있습니다. "구조는 시스템이 잡고, 내용은 사용자가 자유롭게" 접근.

옵션 C 와이어프레임 — 하이브리드
┌──────────────────────────────────────────────┐
│ 10:32 AM · [Note]                [⋯ 메뉴]│
│ Starting payment flow exploration. │
│ Focus: 3DS verification edge cases        │
│───────────────────────────────────────────── │
│ 10:45 AM · [Bug] 🐛 Major         [⋯ 메뉴]│
│ 3DS timeout causes silent failure         │
│ After exactly 30s, payment fails with no │
│ error. Console shows: ERR_TIMEOUT_3DS      │
│ 📷 [payment-error.png] (인라인 썸네일) │
│ 🔗 Linked: JIRA-1234                        │
│───────────────────────────────────────────── │
│ 10:52 AM · [Observation]        [⋯ 메뉴]│
│ Concurrent sessions: both succeed but one │
│ shows wrong currency symbol (€ instead of $) │
├──────────────────────────────────────────────┤
│ What did you find?                          │
│ [+Note] [+Bug🐛] [+Observation👁] [+Step🦶] [+📷]       │
└──────────────────────────────────────────────┘

✅ 장점

  • 구조 + 자유도 모두 확보: 유형/타임스탬프는 자동, 내용은 자유 서식
  • CEO 우려 해결: 유형 선택이 최소한의 구조를 강제함
  • Bug 엔트리는 심각도 필수 → 데이터 품질 보장
  • 리포트 자동 생성: "Bug 3개 (Major 2, Minor 1), Observation 5개" 즉시 집계
  • 각 엔트리 내에서 Bold, 이미지, 코드블록 자유 사용 → 표현력 유지
  • 빠른 입력: 하단 버튼 원클릭 → 유형 자동 선택 + 포커스 이동
  • Notion/Linear의 블록 기반 + Slack의 시간순 메시지 장점 결합

❌ 단점

  • 구현 복잡도 중간 (옵션 A < 옵션 C < 옵션 D)
  • 엔트리 간 드래그앤드롭 재정렬 필요 시 추가 구현

경쟁사 사례: Linear의 이슈 코멘트 (구조화된 메타데이터 + 리치 텍스트 본문), Notion의 블록 기반 편집 (각 블록이 타입을 가짐).

옵션 D: 슬래시 커맨드 블록 에디터 (Notion 스타일)

/ 커맨드로 블록 타입 삽입 — 고급 사용자 최적화

하나의 에디터에서 / 입력 시 블록 타입 메뉴가 나타나고, 원하는 타입을 선택하면 해당 구조의 블록이 삽입됩니다. Notion의 슬래시 커맨드와 동일한 UX.

✅ 장점

  • 파워 유저에게 최고의 속도 (키보드에서 손을 떼지 않음)
  • 블록 타입을 무한 확장 가능 (/bug, /note, /screenshot, /table, /checklist...)
  • 기본 텍스트 작성 시 추가 단계 없음 (타입 지정은 선택)

❌ 단점

  • 학습 곡선 높음: / 커맨드 존재를 모르면 자유 텍스트로만 사용
  • 구조화 강제 불가 — 사용자가 / 안 쓰면 옵션 A와 동일한 문제 발생
  • 구현 복잡도 가장 높음 (블록 에디터 + 커맨드 파서 + 블록 타입 시스템)
  • CEO 우려를 근본적으로 해결 못함 (비기술 사용자는 / 안 씀)

경쟁사 사례: Notion, Coda. 테스트 관리 도구 중에서는 이 방식을 채택한 곳 없음 (과잉 설계 위험).

3-3. 히트맵 대안 — 더 실용적인 Activity 시각화 옵션

대안 1: Activity Timeline (타임라인 바)

히트맵 대신, 세션 시간을 수평 바로 표현하고 각 로그 엔트리를 점(dot)으로 표시. 유형별 컬러 코딩.

히트맵 대신, 세션의 핵심 수치를 카드 형태로 요약 표시. 가장 단순하면서 실용적.

Session Log: 옵션 C (하이브리드 — 구조화 엔트리 + 리치 텍스트)

가중 합산 8.10점으로 최고. CEO의 핵심 우려인 "작성 품질 편차"를 유형 선택 강제로 해결하면서, 테스터의 표현 자유도를 유지합니다. 또한 구조화된 데이터로 리포트 자동 생성, AI 분석, 세션 간 비교가 가능해집니다.

기존 히트맵을 완전 대체. 로그 엔트리 기반 측정으로 구현 난이도를 대폭 낮추면서, 핵심 수치(시간, 엔트리 수, 버그 수) + 시간대별 패턴을 모두 보여줍니다. 히트맵의 모호한 "활동량"을 구체적인 "로그 엔트리 작성 빈도"로 명확화합니다.

6 CEO 피드백 #1 — 엔트리 타입별 입력 확장 폼

6-1. 문제 분석

현재 하단 입력 바는 모든 엔트리 타입에 대해 동일한 1줄 텍스트 입력을 사용합니다. 그러나 엔트리 타입별로 필요한 필드 구조가 다릅니다:

엔트리 타입필요 필드1줄 입력으로 가능?
Note내용 (1개)가능
Observation내용 (1개)가능
Screenshot이미지 + 캡션 (2개, 캡션 선택)가능 (캡션은 선택)
Bug제목 + 내용 + 심각도 + (스크린샷) + (이슈 링크)불가능
StepTested + Expected + Actual (3개 필수)불가능
