1 Discovery Log의 목적과 사용 시나리오 분석

1-1. Discovery Log란?

Discovery Log는 탐색적 테스팅(Exploratory Testing) 세션을 구조화하여 관리하는 Testably의 핵심 탭입니다. 사전에 정의된 Test Case를 실행하는 것과 달리, 테스터가 자유롭게 제품을 탐색하면서 발견한 버그, 이슈, 관찰 사항을 실시간으로 기록합니다.

1-2. 핵심 사용 시나리오

시나리오 A: 타임박스 탐색 세션

QA 엔지니어가 1~2시간의 타임박스를 설정하고, 특정 영역(예: 결제 플로우)을 자유 탐색합니다. 세션 중 발견한 버그를 즉시 기록하고, 세션 종료 시 요약 리포트를 생성합니다.

시나리오 B: 팀 기반 탐색

여러 테스터가 동시에 다른 영역을 탐색합니다. PM/리드는 대시보드에서 진행 상황, 발견 버그 수, 활성 세션을 모니터링합니다.

시나리오 C: 버그 트리아지 연동

탐색 세션에서 발견된 버그를 Jira/Linear 티켓으로 자동 연동하고, 마일스톤과 연결하여 릴리스 판단 근거로 활용합니다.

Testably의 기회: Qase 수준의 UX + TestRail 수준의 기능 + 네이티브 마일스톤/런 연동으로 차별화 가능.

현재 18-discovery-compact.html의 카드 레이아웃을 유지하면서, 2-row 통일 헤더를 적용하고 검색/필터를 추가하는 방안입니다.

✅ 장점 (Pros)

  • 기존 디자인 자산 재활용 → 개발 비용 최소
  • Charter/Mission 텍스트를 자연스럽게 표시 가능
  • 세션의 "스토리" 성격과 카드 UI가 잘 맞음
  • 태그, 마일스톤 등 메타데이터 시각적 표현 용이
  • 모바일/태블릿 반응형 대응 쉬움

❌ 단점 (Cons)

  • 세션이 50개 이상일 때 스크롤 과다 → 정보 밀도 낮음
  • 열 기준 정렬이 불가 (날짜순, 버그 수순 등)
  • 대량 데이터 비교/스캔 시 비효율적
  • 일괄 작업(다중 선택, 상태 일괄 변경) 구현 어려움
  • Testably 다른 탭(TC, Runs)이 테이블인데 일관성 저해

적합 대상: 세션 수가 적고(30개 이하), Charter 중심으로 세션을 관리하는 소규모 팀.

4 옵션 B: 테이블 형태

Test Cases, Runs & Results 탭과 동일한 테이블 레이아웃을 적용합니다. 좌측 사이드바에는 태그 또는 마일스톤 기반 필터 트리를 배치합니다.

✅ 장점 (Pros)

  • 높은 정보 밀도 → 한 화면에 15~25개 세션 표시 가능
  • 열 기준 정렬 (날짜, 버그 수, 상태) 네이티브 지원
  • 다중 선택 + 일괄 작업 자연스러움
  • TC, Runs 탭과 동일한 패턴 → 사용자 학습 비용 제로
  • 좌측 사이드바로 태그/마일스톤 필터링 가능

❌ 단점 (Cons)

  • Charter/Mission 텍스트를 테이블 셀에 표시하기 어려움
  • 세션의 "스토리/컨텍스트" 전달력 약화
  • 태그가 많을 경우 셀 너비 초과 문제
  • 카드 대비 시각적 풍부함 감소

적합 대상: 세션이 많고(50개 이상), 데이터 기반으로 세션을 관리/비교하는 대규모 QA 팀.

5 옵션 C: 하이브리드 (테이블 기본 + 확장 카드)

기본 뷰는 테이블이되, 각 행을 클릭/확장하면 카드 형태의 상세 패널이 인라인으로 펼쳐지는 하이브리드 방식입니다. Qase의 접근법을 참고하되 Testably만의 패턴으로 발전시킵니다.

✅ 장점 (Pros)

  • 테이블의 정보 밀도 + 카드의 컨텍스트 전달력 동시 확보
  • 기본 뷰는 깔끔한 테이블 → 확장 시 풍부한 정보 제공
  • 정렬, 다중 선택, 일괄 작업 모두 가능
  • Charter를 확장 영역에서 전체 표시 가능
  • TC/Runs 탭과 기본 패턴 일치 + Discovery만의 확장 차별화
  • 사이드바 필터 트리 동일 적용 가능

❌ 단점 (Cons)

  • 확장/접기 인터랙션 구현 비용 추가
  • 동시에 여러 행이 확장되면 레이아웃 복잡해질 수 있음
  • 키보드 내비게이션 설계 필요 (확장 상태에서 J/K 동작)

적합 대상: 다양한 규모의 팀 모두. 테이블의 효율성과 카드의 표현력을 모두 원하는 경우.

7-3. 헤더 절약 효과

기존 Discovery Log는 별도 page-head(~56px) + sub-tabs(~40px) = 96px을 헤더에 사용했습니다. 2-row 통일 헤더 적용 시 Row 2(42px)에 서브탭과 액션 버튼을 모두 통합하므로, 약 54px의 수직 공간을 절약합니다. 이는 테이블 뷰에서 약 1.5개 행을 추가로 표시할 수 있는 공간입니다.
