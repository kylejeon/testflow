3
Analytics 탭: 위젯 상세 설계

Analytics 탭은 8개 위젯으로 구성하며, 3단계(Phase)로 나누어 구현합니다. 상단에 공통 기간 필터(7일 / 14일 / 30일 / 전체)와 마일스톤 필터를 배치합니다.

A. Pass Rate 트렌드  P0 · Phase 1
Free+

목적: 시간에 따른 테스트 성공률 변화를 한눈에 파악. QA 팀의 가장 기본적인 건강 지표.

시각화

  • 라인 차트 — X축: 날짜, Y축: Pass Rate %, 보조선: 실행 건수 (바 차트 오버레이)
  • 기간 선택에 따라 일별/주별 그래핑
  • 이전 기간 대비 델타(Δ) 배지 표시 (▲+5.2% 또는 ▼-3.1%)

KPI 카드 (차트 상단 4개)

카드값델타
Overall Pass Rate(passed / executed) × 100vs 이전 동일 기간
Total Executed기간 내 실행된 결과 수vs 이전 기간
Failed실패 건수vs 이전 기간
Blocked차단 건수vs 이전 기간

데이터 소스

// 재사용: usePassRateReport 훅 const { data } = usePassRateReport(projectId, trendPeriod); // test_results WHERE project_id = ? AND created_at >= ? // GROUP BY DATE(created_at)
B. 팀원별 성과 분석  P0 · Phase 1  차별화
Professional+

목적: 팀원별 테스트 실행량, 발견 버그 수, 응답 속도를 비교. 경쟁사 4개 모두 이 기능이 없음.

시각화

  • 팀원 리더보드 테이블 — 이름, 실행 수, 발견 실패 수, 평균 응답 시간, 활동 스파크라인
  • 스택 바 차트 — 팀원별 Passed / Failed / Blocked 분포
  • 정렬 옵션: 실행 수, 실패 발견 수, 응답 속도

메트릭 정의

메트릭계산 방법의미
실행 건수COUNT(test_results) WHERE author = member테스트 처리량
버그 발견율COUNT(failed) / COUNT(executed)높을수록 효과적 탐색적 테스트
평균 응답 시간AVG(first_result_at - run.created_at)Run 배정 후 첫 실행까지 시간
7일 활동 스파크라인일별 COUNT(test_results)최근 활동 패턴

데이터 소스

// test_results.author → profiles.full_name 조인 // GROUP BY author, DATE(created_at) // 재사용: useTeamActivity 훅 확장
C. Flaky TC 탐지 (불안정 테스트)  P1 · Phase 2  AI 차별화
Professional+

목적: 같은 TC가 Pass/Fail을 반복하면 "Flaky"로 자동 탐지. 테스트 신뢰도를 떨어뜨리는 주범을 식별. 경쟁사 전원 미제공.

시각화

  • Flaky Score 테이블 — TC명, 최근 N회 결과 시퀀스(●●○●○), Flaky Score, 마지막 실행일
  • Flaky Score 공식: 최근 10회 실행에서 상태 전환(Pass→Fail, Fail→Pass) 횟수 ÷ 9 × 100
  • Score ≥ 50%를 Flaky로 표시, 빨간색 하이라이트

AI 인사이트 (Pro+)

🤖 AI Flaky 분석: "TC-042 '로그인 세션 만료 테스트'가 지난 2주간 6회 중 3회 Fail → Pass를 반복했습니다. 실패는 모두 월요일 오전에 발생하여, 주말 캐시 만료와 관련될 수 있습니다. 환경 의존적 TC로 분류를 권장합니다."

데이터 소스

// test_results WHERE test_case_id = ? // ORDER BY created_at DESC LIMIT 10 // 전체 TC에 대해 Flaky Score 계산 → 상위 10개 표시 // 신규 쿼리 필요 (기존 훅에 없음)
D. 폴더/모듈별 커버리지 히트맵  P1 · Phase 2  차별화
Free+

목적: 프로젝트의 어떤 모듈/폴더가 테스트가 잘 되어 있고, 어디가 취약한지 한눈에 파악. qTest만 기본 히트맵 제공, 나머지 경쟁사 없음.

시각화

  • 트리맵 히트맵 — 각 폴더를 사각형으로 표시, 크기 = TC 수, 색상 = Pass Rate (녹색 100% ~ 빨강 0%)
  • 호버 시 상세 정보: 폴더명, TC 수, Pass/Fail/Untested 수, Pass Rate %
  • 클릭 시 해당 폴더의 TC 목록으로 드릴다운

데이터 소스

// test_cases.folder → 폴더별 그룹핑 // LEFT JOIN test_results (최신 결과만) → 상태 집계 // GROUP BY folder // 기존 데이터로 계산 가능, 신규 쿼리 필요
E. 마일스톤 트래커 & 릴리즈 번다운  P0 · Phase 1
Free+

목적: 각 마일스톤의 진척도와 기한까지 남은 테스트를 번다운 차트로 시각화. 릴리즈 가능 여부를 데이터 기반으로 판단.

시각화

  • 번다운 차트 — X축: 날짜, Y축: 잔여 미실행 TC 수. 이상선(ideal line) vs 실제선
  • 마일스톤 카드 그리드 — 각 마일스톤별 진행률 도넛, 남은 일수, 상태 배지(on track / at risk / overdue)
  • Overview 탭의 Release Readiness Score와 연계: Score ≥80 "Go", 60-79 "Conditional", <60 "Not Ready"

위험 판정 로직

// daysRemaining = milestone.end_date - today // remainingTC = untested TC count in linked runs // dailyVelocity = executed TC count / elapsed days // estimatedDaysToComplete = remainingTC / dailyVelocity // // if (estimatedDaysToComplete > daysRemaining) → "At Risk" 🟠 // if (end_date < today && progress < 100) → "Overdue" 🔴 // else → "On Track" 🟢

데이터 소스

// milestones WHERE project_id = ? AND status IN ('upcoming','started') // test_runs WHERE milestone_id = ? // test_results WHERE run_id IN (linked runs) // 기존 데이터 활용, 번다운 계산 로직 신규
F. TC 품질 & 성장 분석  P1 · Phase 2
Free+

목적: TC 자산의 건강 상태를 한눈에 파악 — 성장 추이, Priority 분포, 자동화율, 가장 많이 실패하는 TC.

시각화

  • TC 성장 에어리어 차트 — 주별 누적 TC 수 (재사용: useTestCasesOverview)
  • Priority 도넛 차트 — Critical / High / Medium / Low 분포
  • Lifecycle 상태 바 — Active / Draft / Deprecated 비율
  • 자동화율 게이지 — is_automated = true 비율
  • Top 10 실패 TC 테이블 — 실패 횟수 순 정렬, Priority 배지, 마지막 실패일

데이터 소스

// 재사용: useTestCasesOverview 훅 // + test_results GROUP BY test_case_id, status // HAVING COUNT(status='failed') ORDER BY DESC LIMIT 10
G. AI 인사이트 패널  P2 · Phase 3  핵심 차별화
Professional+

목적: AI가 프로젝트 데이터를 분석하여 자연어 인사이트와 액션 추천을 제공. 경쟁사 전원 미제공 — Testably만의 핵심 차별화.

🤖 AI-Native 포지셔닝의 핵심
단순한 차트는 모든 경쟁사가 제공합니다. "데이터를 보여주는 것"이 아니라 "데이터가 의미하는 바를 알려주는 것"이 Testably의 차별화입니다.

AI 인사이트 유형

인사이트 유형예시데이터 기반
일일 요약 "오늘 팀이 142건의 TC를 실행했고, Pass Rate가 어제 대비 3.2% 상승했습니다. 특히 '결제 모듈' 폴더의 Pass Rate가 95%로 안정화되었습니다." test_results (당일), 폴더별 집계
완료 예측 "현재 속도(일 평균 45건)로 진행하면, Sprint 12 마일스톤은 4월 8일에 완료 예상됩니다. 기한(4월 10일)보다 2일 여유가 있습니다." milestones, daily velocity 계산
리스크 경고 "'사용자 인증' 폴더의 Pass Rate가 지난 3일간 78% → 65% → 52%로 급락 중입니다. 최근 코드 변경의 영향일 수 있으니 확인이 필요합니다." 폴더별 Pass Rate 3일 트렌드
팀 밸런스 "김민수 님에게 미실행 TC 38건이 집중되어 있습니다. 이정현 님(현재 유휴)에게 12건을 재배정하면 3일 단축할 수 있습니다." assignee별 미실행 TC, 팀원별 velocity
품질 트렌드 "이번 주 새로 발견된 결함 18건 중 12건(67%)이 '결제' 모듈에 집중되어 있습니다. 해당 모듈에 리그레션 테스트를 추가로 수행하는 것을 추천합니다." test_results(failed), folder 분포

구현 방식

// 1. 프론트엔드에서 분석 데이터 수집 (이미 fetch된 데이터 활용) // 2. 규칙 기반 인사이트 (Phase 3a): 임계값 기반 경고/요약 생성 // - Pass Rate Δ > 10% → 급변 경고 // - Flaky Score > 50% TC 존재 → Flaky 경고 // - remainingTC / velocity > daysRemaining → 기한 위험 // 3. LLM 기반 인사이트 (Phase 3b): Supabase Edge Function으로 // 집계 데이터를 LLM에 전달, 자연어 요약 생성 // → 티어 게이팅: Professional+ 전용

Activity Feed 탭: 상세 설계

Activity Feed는 프로젝트의 모든 활동을 실시간으로 추적하는 "프로젝트의 맥박"입니다. 경쟁사 4개 모두 실시간 Activity Feed가 없으며, Supabase Realtime을 활용하면 즉시 차별화됩니다.

4.1 트래킹 대상 활동 유형

카테고리이벤트표시 예시데이터 소스Phase

4.4 실시간 업데이트 (Supabase Realtime)

Supabase Realtime 활용: Supabase의 Realtime 기능으로 DB 변경 사항을 WebSocket으로 즉시 수신할 수 있습니다. Activity Feed에 실시간 푸시를 적용하면, 팀원이 TC를 실행하는 순간 다른 팀원의 화면에 즉시 반영됩니다.
// Activity Feed 실시간 구독 예시 import { supabase } from '../../lib/supabase'; useEffect(() => { const channel = supabase .channel('project-activity') .on('postgres_changes', { event: 'INSERT', schema: 'public', table: 'test_results', filter: `project_id=eq.${projectId}`, }, (payload) => { // 새 결과가 추가되면 피드 상단에 즉시 표시 addFeedItem(formatTestResult(payload.new)); }) .on('postgres_changes', { event: '*', schema: 'public', table: 'test_cases', filter: `project_id=eq.${projectId}`, }, (payload) => { addFeedItem(formatTestCaseChange(payload)); }) .subscribe(); return () => { supabase.removeChannel(channel); }; }, [projectId]);

4.6 DB 스키마 — activity_logs 테이블 (신규)

아키텍처 결정 필요: 현재 Activity 데이터는 여러 테이블에서 JOIN으로 조합합니다. 성능과 확장성을 위해 전용 activity_logs 테이블을 도입하는 것을 권장합니다.
-- 옵션 A: 전용 activity_logs 테이블 (추천) CREATE TABLE activity_logs ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE, actor_id UUID REFERENCES auth.users(id), event_type TEXT NOT NULL, -- 'test_result_created', 'tc_created', 'run_started', ... event_category TEXT NOT NULL, -- 'test_execution', 'tc_management', 'milestone', 'team', 'jira' target_type TEXT, -- 'test_result', 'test_case', 'test_run', 'milestone' target_id UUID, -- 대상 레코드 ID metadata JSONB DEFAULT '{}', -- 이벤트별 추가 데이터 (TC명, 상태, Run명 등) is_highlighted BOOLEAN DEFAULT false, -- 중요 이벤트 플래그 created_at TIMESTAMPTZ DEFAULT now() ); -- 인덱스 CREATE INDEX idx_activity_logs_project ON activity_logs(project_id, created_at DESC); CREATE INDEX idx_activity_logs_actor ON activity_logs(actor_id, created_at DESC); CREATE INDEX idx_activity_logs_category ON activity_logs(event_category, created_at DESC); -- RLS ALTER TABLE activity_logs ENABLE ROW LEVEL SECURITY; CREATE POLICY "Users can view activity in their projects" ON activity_logs FOR SELECT USING ( project_id IN (SELECT project_id FROM project_members WHERE user_id = auth.uid()) );

7.3 DB 트리거 — activity_logs 자동 기록

-- test_results INSERT → activity_logs 자동 기록 CREATE OR REPLACE FUNCTION fn_log_test_result() RETURNS TRIGGER AS $$ BEGIN INSERT INTO activity_logs (project_id, actor_id, event_type, event_category, target_type, target_id, metadata, is_highlighted) SELECT tr.project_id, (SELECT author_id FROM ... ), -- author UUID 매핑 필요 'test_result_' || NEW.status, 'test_execution', 'test_result', NEW.id, jsonb_build_object( 'tc_title', tc.title, 'tc_custom_id', tc.custom_id, 'status', NEW.status, 'run_name', run.name ), CASE WHEN tc.priority = 'critical' AND NEW.status = 'failed' THEN true ELSE false END FROM test_cases tc JOIN test_runs run ON run.id = NEW.run_id WHERE tc.id = NEW.test_case_id; RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER trg_log_test_result AFTER INSERT ON test_results FOR EACH ROW EXECUTE FUNCTION fn_log_test_result();

유사한 트리거를 test_cases, test_runs, milestones, project_members 테이블에도 추가합니다.
