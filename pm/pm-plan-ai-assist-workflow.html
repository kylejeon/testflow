Executive Summary

현재 Testably에는 두 개의 독립된 AI 생성 시스템이 존재합니다:

  1. AIAssistModal (Project Detail Dashboard) — 5개 액션 허브, 자체 TC 생성/저장 로직, Edge Function generate-test-cases 호출
  2. AIGenerateModal (Test Cases 페이지) — 3탭 위저드(Plain Text / Jira / Discovery), Edge Function generate-testcases 호출

핵심 문제: 두 모달이 서로 다른 Edge Function을 호출하며, 동일한 "Generate Test Cases" 기능을 중복 구현하고 있습니다. 또한 3개 액션(Summarize Run, Analyze Failed, Coverage Gap)은 comingSoon: true로 비활성 상태입니다.

2. 핵심 문제점

🔴 Issue 1: 중복 구현 — 동일 기능, 다른 백엔드
"Generate Test Cases" 기능이 두 곳에서 서로 다른 Edge Function(generate-test-cases vs generate-testcases)을 호출합니다. TC 저장 방식도 다릅니다 (프론트엔드 직접 INSERT vs Edge Function 서버 사이드). 같은 기능인데 동작이 다를 수 있어 사용자 혼란을 유발합니다.
🔴 Issue 2: AIAssistModal의 Mock Fallback
AIAssistModal.tsx의 handleGenerate 함수에서 Edge Function 호출 실패 시 하드코딩된 Mock TC 6개를 표시합니다 (line 185–192). 사용자가 AI가 실제로 생성한 것으로 착각할 수 있으며, 이 Mock TC를 프로젝트에 저장할 수도 있습니다.
🟡 Issue 3: Jira/Discovery 탭이 UI만 존재
AIAssistModal의 generate 모드에 text/jira/discovery 3탭이 표시되지만, 실제로는 source: activeTab만 Edge Function에 전달할 뿐 입력 UI는 동일한 textarea입니다. Jira Issue Key 입력이나 Discovery 세션 선택 같은 특화 UI가 없습니다.
🟡 Issue 4: 3개 Coming Soon 액션에 대한 계획 부재
Summarize Run Results, Analyze Failed Tests, Coverage Gap Analysis가 comingSoon: true로 비활성화되어 있으나, 어떤 Edge Function이 필요한지, 입출력 스펙이 무엇인지 정의되어 있지 않습니다.

Summarize Run Results

미구현

기능 목적

선택한 Test Run의 결과를 AI가 분석하여 실행 요약 리포트를 자동 생성합니다. QA 매니저/PM이 빠르게 현황을 파악할 수 있게 합니다.

워크플로우

AI Assist → "Summarize Run" 클릭 → Run 선택 드롭다운 → "Summarize" 버튼 → AI 분석 중... → 요약 리포트 표시

입력 데이터

필수Test Run ID (드롭다운에서 선택)
자동 수집해당 Run의 모든 TC 결과 (test_run_results 테이블): 각 TC의 title, status(pass/fail/skip/block), 실행 시간, 코멘트
컨텍스트Run 이름, 생성일, 전체 Pass Rate, 실행자 정보

출력 — AI 생성 요약

Overall StatusPass/Fail/Skip 비율 요약 (시각적 바 차트)
Key Findings실패 패턴 3~5개 (예: "인증 관련 3개 TC 모두 실패 — 로그인 서비스 문제 의심")
Risk Assessment릴리스 리스크 레벨 (🟢 Low / 🟡 Medium / 🔴 High)
Recommendations다음 액션 2~3개 제안 (예: "Critical 실패 2건 수정 후 재실행 권장")
Blocked/Skipped AnalysisSkip/Block된 TC가 왜 실행되지 않았는지 분석

}
// → 서버에서 test_run_results를 조회하여 AI에 전달

  "summary": {

    "recommendations": ["...", "..."],

UI 모킹

요약 결과를 카드 형태로 표시합니다. 상단에 Pass Rate 게이지, Key Findings를 불릿 리스트, Risk Assessment를 컬러 배지로, Recommendations를 체크리스트 형태로 표시합니다. "Copy to Clipboard" 버튼으로 Slack/이메일에 붙여넣기 가능.

Analyze Failed Tests

미구현

기능 목적

프로젝트의 실패한 TC들을 AI가 분석하여 실패 패턴을 식별하고, 우선순위별 수정 권장 사항을 제공합니다.

워크플로우

AI Assist → "Analyze Failed" 클릭 → 범위 선택 (최근 Run / 전체 프로젝트 / 특정 Run) → "Analyze" 버튼 → AI 분석 중... → 분석 결과

입력 데이터

범위 선택① 최근 Run의 실패 TC만 ② 전체 프로젝트의 실패 TC ③ 특정 Run 선택
자동 수집실패 TC 목록: title, module/folder, priority, 실패 횟수(히스토리), 실행 코멘트, 마지막 실패 일시

출력 — AI 분석 결과

Failure Clusters관련 실패를 그룹핑 (예: "모듈 A의 3건, API 응답 관련 4건")
Root Cause Hypotheses각 클러스터의 추정 원인 (예: "인증 토큰 만료 → 3건의 401 에러")
Priority Ranking수정 우선순위: 🔴 Critical → 🟡 Medium → 🟢 Low
Flaky Test Detection같은 TC가 pass/fail 반복하는 패턴 감지
Fix Suggestions각 클러스터에 대한 수정 제안 (2~3줄)

Edge Function 스펙

// POST /functions/v1/ai-analyze-failures
{
  "project_id": "uuid",
  "scope": "latest_run" | "all" | "specific_run",
  "run_id": "uuid" // (scope: specific_run일 때만)
}
// Response:
{
  "analysis": {
    "total_failures": 8,
    "clusters": [
      {
        "name": "Authentication Module Failures",
        "test_cases": ["TC-001", "TC-003", "TC-007"],
        "root_cause": "Session token expiration not handled",
        "priority": "critical",
        "suggestion": "Add token refresh logic before API calls"
      }
    ],
    "flaky_tests": [
      { "title": "Cart persistence test", "pass_rate_30d": 60, "pattern": "intermittent" }
    ]
  }
}

UI 디자인

각 Failure Cluster를 확장 가능한 카드로 표시. 카드에는 TC 목록, 추정 원인, 수정 제안이 포함됩니다. 우선순위별 색상 코딩 (Critical=빨강, Medium=주황, Low=초록). Flaky Tests는 별도 섹션에 경고 배지로 표시.

Coverage Gap Analysis

미구현

기능 목적

프로젝트의 기존 TC를 AI가 분석하여 테스트되지 않은 영역을 식별하고, 추가 TC를 자동 제안합니다.

워크플로우

AI Assist → "Coverage Gap" 클릭 → 분석 범위 선택 (전체 / 특정 폴더·모듈) → "Analyze" 버튼 → AI 분석 중... → 갭 분석 + TC 제안

입력 데이터

범위 선택① 전체 프로젝트 ② 특정 폴더/모듈만
자동 수집모든 TC: title, module/folder, type(functional/smoke/regression/edge_case), priority, status
선택 입력프로젝트 설명 또는 기능 목록 (Coverage 정확도 향상용)

출력 — AI 분석 결과

Coverage Map모듈/폴더별 TC 수와 타입 분포 (시각적 히트맵)
Identified Gaps테스트 커버리지 부족 영역 리스트 (예: "에러 처리 관련 TC 0건", "모바일 뷰 TC 부족")
Suggested TCs각 갭에 대한 추가 TC 제안 (체크박스로 선택 → 바로 프로젝트에 추가)
Type BalanceTC 타입 비율 분석 (Functional 80%, Edge Case 5% → "Edge Case 비율 낮음" 경고)
Priority Areas커버리지 확대 우선순위 (High-priority 모듈 중 TC가 적은 영역)

UI 디자인

상단에 모듈별 히트맵 (색상 = 커버리지 점수), 중앙에 Identified Gaps 카드 리스트 (각 카드에 제안 TC 포함), 하단에 Type Balance 도넛 차트. Suggested TCs에 체크박스가 있어 선택 후 "Add to Project" 가능 — 이 저장 로직은 Edge Cases와 동일한 패턴 재사용.
