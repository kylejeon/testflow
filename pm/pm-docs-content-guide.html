Test Cases
Test cases are the building blocks of your QA process. Learn how to create, organize, and manage test cases effectively — manually, in bulk, or with AI assistance.

Section 1: Creating Test Cases

1-1. 수동 생성 (Manual Creation)

콘텐츠 텍스트

To create a new test case, navigate to your project's Test Cases tab and click the "+ New Test Case" button in the top-right corner. This opens the test case editor where you can fill in the following fields:

1-2. AI 생성 (AI-Powered Generation)

콘텐츠 텍스트

Testably can automatically generate test cases using AI. Click the "AI Generate" button to open the AI generation wizard.

Two generation modes are available:

Mode 1 — Text-based: Describe a feature or requirement in natural language, and AI will generate relevant test cases. Example: "User registration with email, Google SSO, and password validation."

Mode 2 — Session-based (Professional+): Select an existing Discovery Log session, and AI will analyze the session's observations to generate structured test cases.

The generation flow has 4 steps:

  1. Choose mode — Text-based or Session-based
  2. Provide input — Type feature description or select a session
  3. Review titles — AI generates suggested test case titles. Edit, remove, or add more.
  4. Review details — AI fills in full details (steps, expected results, priority). Edit as needed, then save.
Plan Limits — AI Generation
Free: 5 generations/month  |  Starter: 30/month  |  Professional: 150/month  |  Enterprise: Unlimited
Session-based mode requires Professional plan or above.

Section 2: Organizing with Folders

콘텐츠 텍스트

Folders help you organize test cases into logical groups. You can structure them by feature, module, sprint, or any system that fits your team.

Creating a folder: When creating or editing a test case, type a new folder name in the Folder field — it will be created automatically. Folders can have custom icons and colors for visual distinction.

Available folder colors: Indigo, Violet, Pink, Emerald, Amber, Cyan, Red, Teal, Orange, Blue (10 colors).

Sidebar navigation: The test case list shows a left sidebar with all folders. Click a folder to filter test cases. The count next to each folder shows how many test cases it contains.

Moving test cases to folders: Select one or more test cases → use the Bulk Action Bar's "Move" button → choose the target folder.

Section 5: Editing Test Case Details

콘텐츠 텍스트

Click any test case to open its detail view. Here you can edit all fields, add steps, and manage attachments.

Steps & Expected Results: Break down your test into numbered steps. Each step describes a single user action. The expected result describes what should happen after all steps are completed successfully.

Step format example:

  1. Navigate to the login page
  2. Enter valid email address
  3. Enter valid password
  4. Click the "Sign In" button

Expected Result: User is redirected to the dashboard and sees a welcome message.

Attachments: You can attach screenshots, documents, or any files to a test case for additional context.

Lifecycle transitions: Move a test case through its lifecycle: Draft (initial creation) → Active (ready for test runs) → Deprecated (no longer relevant). Deprecated cases are hidden by default but can be viewed using the lifecycle filter.

Tip
Write test case titles as user actions: "User can [action]" or "Verify that [behavior]". This makes test cases self-documenting and easier to understand during execution.

Test Runs
A test run is a single execution of a set of test cases. Create runs tied to milestones, assign test cases to team members, and execute them in Focus Mode for a distraction-free testing experience.

Section 1: Creating a Test Run

콘텐츠 텍스트

Navigate to your project's Test Runs tab and click "+ New Run".

1
Name your run
Give the run a descriptive name (e.g., "Sprint 12 Regression", "Release 2.5 Smoke Test").
2
Link to a Milestone (optional)
Select a milestone to associate this run with a release or sprint target. This contributes to the milestone's progress tracking.
3
Select Test Cases
Choose which test cases to include. You can select individually, by folder, or select all. The run shows a count of selected cases.
4
Assign to team members (optional)
Assign specific test cases to team members. Each assignee sees their assigned cases when they open the run.
5
Create the run
Click "Create" to start the run. Status begins as New.

Automated runs (CI/CD): Runs created through the CI/CD API are marked with an automation badge. These runs receive results from your pipeline automatically.

Section 3: Assigning Test Cases

콘텐츠 텍스트

Within a test run, each test case can be assigned to a specific team member. This is useful for distributing work across the QA team.

How to assign: Open the run detail page → click the assignee icon next to any test case → select a team member from the dropdown. Only project members with Owner, Admin, or Member roles appear in the assignee list.

Filtering by assignee: On the run detail page, use the contributor filter to see only test cases assigned to a specific person.

Section 4: Focus Mode

콘텐츠 텍스트

Focus Mode is Testably's distraction-free testing interface. It's designed for efficient test execution with everything you need on one screen.

Layout:

  • Left panel: List of all test cases in the run. Shows status indicators (passed/failed/blocked/retest/untested) for each case. Click a case to load it.
  • Right panel: The active test case's full details — title, steps, expected result, preconditions, and attachments. Below is the result input area.

To enter Focus Mode: Open a test run → click the "Focus Mode" button (or click any individual test case). The URL includes a ?focus=true parameter.

Section 5: Recording Results

콘텐츠 텍스트

In Focus Mode, record the result of each test case execution:

StatusIconWhen to Use
Passed✅ GreenThe test case executed as expected. All steps completed successfully.
Failed❌ RedThe actual result differs from the expected result. A defect exists.
Blocked⛔ YellowThe test cannot be executed due to a dependency, environment issue, or prerequisite failure.
Retest🔄 BlueA previously failed test needs to be re-executed after a fix.
Untested⬜ GrayThe test has not been executed yet (default state).
콘텐츠 텍스트 (계속)

Additional result fields:

  • Note: Add a comment explaining the result (e.g., "Failed on step 3 — button not clickable").
  • Elapsed time: Record how long the test took to execute.
  • Step-level status: Mark individual steps as passed or failed for granular tracking.
  • Issues: Link Jira issues directly to failed test results (requires Jira integration).
  • Attachments: Attach screenshots or evidence files to the result.

Run progress: As you record results, the run's progress bar updates in real-time. The run detail page shows a breakdown: X passed, Y failed, Z blocked, W retest, V untested.

Tip
Use keyboard shortcuts in Focus Mode for faster execution. Navigate between test cases and record results without touching the mouse.

Milestones
Milestones represent release targets or sprint goals. Link test runs to milestones to track testing progress toward a release, and use sub-milestones to break large releases into manageable phases.

Section 1: Creating a Milestone

콘텐츠 텍스트

Navigate to your project's Milestones tab and click "+ New Milestone".

Required fields:

  • Name: Descriptive name (e.g., "Release v2.5", "Sprint 12", "MVP Launch")
  • Start Date: When testing for this milestone begins
  • End Date: Target completion date

Once created, the milestone starts in Upcoming status and transitions automatically based on dates and progress.

Milestone Status Flow

Overduepast_dueEnd date has passed but not all tests are completeRed badge

📸 스크린샷 필요
Milestones 목록 페이지 — 여러 상태의 마일스톤 카드들 + 진행률 바

Section 2: Linking Test Runs

콘텐츠 텍스트

Link test runs to a milestone to track testing progress. You can link a run when creating it (by selecting a milestone in the creation form) or afterward from the run detail page.

How progress is calculated: The milestone's progress is based on the combined results of all linked test runs. Specifically, it shows the ratio of test cases with a final result (passed, failed, blocked) to the total number of test cases across all linked runs.

Milestone detail page shows:

  • Overall progress bar with passed/failed/blocked breakdown
  • All linked test runs with individual progress
  • Linked discovery sessions
  • Activity timeline
  • Failed and blocked test cases listed for quick triage
  • Jira issues linked from failed test results

Section 3: Sub-Milestones

콘텐츠 텍스트

For large releases, you can create sub-milestones to organize testing into phases.

Creating a sub-milestone: On the Milestones page, click the "+ Sub" button on an existing milestone. The sub-milestone is visually nested under its parent and inherits the parent's timeline context.

Example structure:

  • Release v3.0 (parent)
    • Phase 1: Core Features
    • Phase 2: Integrations
    • Phase 3: Performance Testing

Parent-child relationship: The parent milestone's progress reflects the aggregate progress of all its sub-milestones. You can expand/collapse sub-milestones in the list view.

Section 4: Progress Tracking Dashboard

콘텐츠 텍스트

The Milestone Detail page serves as a progress dashboard for your release:

Key metrics displayed:

  • Overall progress bar: Visual percentage with color-coded segments (green=passed, red=failed, yellow=blocked)
  • Test summary: Total tests, passed count, failed count, blocked count
  • Time tracking: Days remaining, days elapsed, overdue indicator
  • Run contributors: Avatar stack showing all team members involved in the milestone's test runs

Activity timeline: Shows the most recent test results, run completions, and status changes in chronological order.

Failed & Blocked list: A dedicated section lists all test cases that failed or are blocked, grouped by run name, with the author and timestamp — making it easy to prioritize fixes.

Section 5: Release Management

콘텐츠 텍스트

Best practices for using milestones as release gates:

  1. Create a milestone per release — Name it after the release version (e.g., "v2.5.0").
  2. Set realistic dates — Align start/end dates with your sprint or release cycle.
  3. Create focused test runs — Separate regression, smoke, and feature-specific runs. Link all to the milestone.
  4. Monitor the dashboard — Check progress daily. Triage failed and blocked cases promptly.
  5. Complete the milestone — Once all critical tests pass and you're confident in the release, mark the milestone as Completed.

Overdue milestones automatically change to Overdue status and trigger notifications (if configured) to alert the team.

Notification Events
Milestone-related webhook events: milestone_started, milestone_completed, milestone_past_due.
Configure these in Settings → Integrations to get Slack/Teams alerts.

Page Hero 텍스트
Discovery Logs
Discovery Logs are Testably's tool for exploratory testing. Start a session, freely explore your application, and record observations as you go — notes, bugs, and test steps. Later, convert your findings into formal test cases.

Section 1: Starting an Exploratory Session

콘텐츠 텍스트

Navigate to your project's Sessions tab and click "+ New Session".

1
Name your session
Give it a descriptive name that reflects what you're exploring (e.g., "Checkout Flow Exploration", "Mobile Navigation Edge Cases").
2
Define the mission (Charter)
Write a brief charter describing what you plan to explore and what you're looking for. Example: "Explore the checkout flow for edge cases: empty cart, expired coupon, address validation."
3
Set time estimate (optional)
Set a time box for the session (e.g., 30 minutes, 1 hour). A live timer tracks your session duration.
4
Assign participants (optional)
Assign one or more team members to the session. Multiple assignees can contribute observations.
5
Link to milestone (optional)
Associate the session with a milestone for progress tracking.
6
Start exploring
Click "Create" and the session begins. A live timer starts counting. You can pause and resume at any time.

세션 생성 모달/폼 — 이름, 미션, 시간 설정, assignee 영역

Section 2: Recording Observations

콘텐츠 텍스트

Once a session is active, you can record observations in real-time. Each entry is timestamped and added to a chronological timeline.

Four types of entries:

TypeIconPurposeDev1 참고
Step🧪 PurpleRecord a test step or observed behavior that works correctly. Use for documenting what you tested.type: 'passed', label: 'Step'
Bug🐛 RedRecord a defect or unexpected behavior found during exploration.type: 'failed', label: 'Bug'
Blocked⛔ AmberRecord a scenario you couldn't test due to an environment or dependency issue.type: 'blocked'
Note📝 IndigoGeneral observations, improvement ideas, or questions to investigate later.type: 'note'

Adding an entry:

  1. Select the entry type from the bottom toolbar (Step / Bug / Blocked / Note)
  2. Write your observation in the rich text editor (supports formatting, code blocks, lists)
  3. Optionally add screenshots or attachments
  4. Press Enter or click Submit — the entry appears in the timeline

Rich text editor: The observation editor supports full rich text formatting powered by Quill. You can use bold, italic, lists, code blocks, and more.

Editing entries: Click the edit icon on any existing entry to modify its content or type after submission.

Section 3: Timeline View

콘텐츠 텍스트

All session entries are displayed in a chronological timeline. Each entry shows:

  • Type indicator: Color-coded icon (Step/Bug/Blocked/Note)
  • Content: The observation text with rich formatting
  • Timestamp: When the entry was recorded
  • Attachments: Any screenshots or files attached
  • Issues: Linked Jira issues (if any)

Live timer: At the top of the session detail, a live timer shows elapsed time in HH:MM:SS format. The timer pauses when the session is paused and stops when completed.

Section 4: About Section (Session Metadata)

콘텐츠 텍스트

The right sidebar (or header area) shows session metadata:

  • Status: Active (green, timer running) / Paused (yellow, timer stopped) / Completed (gray, session ended)
  • Mission: The charter or goal of the exploration
  • Assignees: Team members participating in the session (avatar display)
  • Time Box: The estimated duration set during creation
  • Duration: Actual elapsed time (excludes paused time)
  • Milestone: Linked milestone (if any)
  • Tags: Session tags for categorization

Session controls:

  • Pause/Resume: Pause the timer during breaks. Paused duration is tracked separately.
  • Complete: End the session. Timer stops, status changes to Completed.
  • Edit: Modify session name, mission, assignees, or milestone after creation.

Section 5: Converting to Test Cases

콘텐츠 텍스트

The most powerful feature of Discovery Logs is converting your exploratory findings into formal test cases.

Manual conversion: From a session entry, click the "Convert to Test Case" action. This creates a new test case with the entry's content as the description/steps.

AI-assisted conversion (Professional+): Use the AI Generate feature with "Session-based" mode. Select a completed session, and AI will analyze all entries to generate structured test cases with proper steps, expected results, and priorities.

Best Practice
Run a 30-minute exploratory session on a new feature, record all observations, then use AI to generate test cases from the session. This is the fastest way to build comprehensive test coverage for a new feature.

Team & Permissions
Collaborate with your team by inviting members to your projects. Testably offers four roles — Owner, Admin, Member, and Viewer — to control who can do what. The project creator is automatically assigned the Owner role. When inviting new members, you can assign Admin, Member, or Viewer. Viewers are always free, making it easy to include stakeholders without increasing costs.

1

Opens the invitation modal.
2
Enter email address
Type the email of the person you want to invite. They don't need a Testably account yet — one will be created when they accept.
3
Choose a role
Select Admin, Member, or Viewer. The role determines what the person can do in the project (see role comparison below).
4
Send invitation
Click "Invite". The person receives an email invitation with a link to join the project.
5
Invitation accepted
When the person clicks the invitation link, they join the project with the assigned role. Their status changes from "Pending" to "Active" in the members list.

콘텐츠 텍스트

Testably has four roles. The Owner role is automatically assigned to the project creator and cannot be assigned via invitation. Admin, Member, and Viewer are available when inviting new members. Each role builds on the previous one:

Owner (보라 뱃지): The project creator is automatically assigned the Owner role. Has all Admin permissions plus exclusive rights: delete the project, transfer ownership. Only one Owner per project. Cannot be assigned via invitation — only through ownership transfer.

Admin (오렌지 뱃지): Full management access. Can invite/remove members, change roles, configure settings and integrations. Can do everything except delete the project or transfer ownership. Best for: team leads, senior QA managers.

Member (인디고 뱃지): Can create and edit test cases, run tests, record results, and use all QA features. Cannot manage team or project settings. Best for: QA engineers, developers who write/execute tests.

Viewer (회색 뱃지): Read-only access. Can view test cases, runs, results, milestones, and sessions — but cannot create or modify anything. Best for: project managers, product owners, stakeholders who need visibility.

Section 3: Access Control Rules

콘텐츠 텍스트

Project-level access: Members are invited per project. A person can have different roles in different projects (e.g., Admin in Project A, Viewer in Project B).

Changing roles: Owners and Admins can change a member's role at any time from the Members section. Select a new role from the dropdown next to the member's name (options: Admin, Member, Viewer). The Owner's role cannot be changed via dropdown. The change takes effect immediately.

Removing members: Owners and Admins can remove members from a project. The member loses access immediately. Their previously recorded test results and contributions remain in the project.

Self-removal: Any member can leave a project by themselves (removing their own access). This is useful when someone transitions off a project.

Section 4: Viewer Free Policy

콘텐츠 텍스트

Viewers are always free — they don't count toward your plan's seat limit. This means you can invite unlimited stakeholders, managers, and executives to view testing progress without any additional cost.

How seat counting works:

  • Your plan has a seat limit (e.g., Free = 1 seat, Starter = 5 seats, Professional = 15 seats)
  • Only Owner, Admin, and Member roles count as "seats"
  • Viewer roles are excluded from the seat count entirely
  • When inviting a new member, the system checks the seat limit — but only against non-Viewer members

Why this matters: Include your entire team — developers, product managers, executives — in the testing process without worrying about plan costs. Viewers get full read access to stay informed about quality status.

Example
On the Starter plan (5 seats), you could have: 1 Owner + 1 Admin + 3 Members + unlimited Viewers. Your entire company can observe testing progress.
