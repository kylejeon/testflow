Issue 1: GitLab 토큰 관리 기능

CEO 보고 이슈 GitLab 지원 API & Tokens 탭 수정

1.1 현재 상태 분석

Settings → API & Tokens 탭에서 CEO가 보고한 문제:

CEO 보고 내용
"GitHub 토큰 정보만 있고, GitLab은 클릭이 안 됨"

현재 코드 분석 결과

영역 GitHub GitLab 문제
CI/CD Pipelines 탭 → 플랫폼 선택 GitHub Actions YAML 스니펫 정상 표시 GitLab CI YAML 스니펫 정상 표시 정상 동작
API & Tokens 탭 → API Tokens 카드 GitHub/GitLab 구분 없이 동일한 testably_* 토큰. 생성/복사/삭제 정상 동작 정상 동작
API & Tokens 탭 → Token Scope Guide GitHub 버튼 활성 (dark 스타일). repo, workflow, read:org 스코프 표시 GitLab 버튼 있으나 클릭해도 아무 변화 없음. 상태 전환 로직 미구현 버그: GitLab 클릭 불가
근본 원인
page.tsx의 Token Scope Configuration Guide 섹션에서 GitHub/GitLab 버튼은 UI만 존재하고, GitLab 클릭 시 콘텐츠를 전환하는 상태(selectedScopeGuide 등)가 없습니다. GitHub 스코프 내용이 하드코딩되어 있어 GitLab 전환이 불가능합니다.

1.2 수정 사항: GitLab Token Scope Guide 구현

A. 상태 관리 추가

page.tsx에 새로운 상태를 추가합니다:

// 기존 selectedPlatform은 CI/CD 탭의 코드 스니펫용 (github | gitlab | python) // 새로 추가: Token Scope Guide의 GitHub/GitLab 전환용 const [scopeGuideTab, setScopeGuideTab] = useState<'github' | 'gitlab'>('github');

B. GitLab/GitHub 스코프 데이터

GitHub (기존 — 유지)
repoRepository read access (for private repos)Required
workflowGitHub Actions workflow executionRequired
read:orgRead organization infoOptional

Recommended: 자동 제공되는 GITHUB_TOKEN 사용이 가장 안전

GitHub Token Docs →

◆ GitLab (신규 추가)
apiFull API access (read/write)Required
read_apiRead-only API accessMinimum
read_repositoryRepository read accessOptional

Recommended: GitLab CI/CD에서 자동 제공되는 CI_JOB_TOKEN 사용. 별도 PAT 불필요

GitLab Token Docs →

C. 구현 방법 (Dev2 가이드)

단계 파일 변경 내용
1 page.tsx (상단 상태) scopeGuideTab state 추가: useState<'github' | 'gitlab'>('github')
2 page.tsx (Token Scope Guide 버튼) GitLab 버튼에 onClick={() => setScopeGuideTab('gitlab')} 추가. 활성 스타일 조건부 적용: scopeGuideTab === 'gitlab' ? 'bg-[#FC6D26] text-white ...' : 'bg-white ...'
3 page.tsx (스코프 데이터) 스코프 배열을 scopeGuideTab에 따라 분기:
github: [{scope:'repo', ...}, {scope:'workflow', ...}, {scope:'read:org', ...}]
gitlab: [{scope:'api', ...}, {scope:'read_api', ...}, {scope:'read_repository', ...}]
4 page.tsx (하단 안내 메시지) GitHub: "Recommended: Using the automatically provided GITHUB_TOKEN..."
GitLab: "Recommended: Using the automatically provided CI_JOB_TOKEN within GitLab CI/CD pipelines is the safest approach — no separate PAT creation needed."
5 page.tsx (외부 링크) GitHub: https://docs.github.com/en/authentication/.../personal-access-tokens
GitLab: https://docs.gitlab.com/ee/user/profile/personal_access_tokens.html
예상 작업량
순수 UI 로직 변경만 필요 (상태 1개 추가 + 조건부 렌더링). 0.5일 이내 완료 가능.
API Tokens 카드 자체는 GitHub/GitLab 공용이므로 변경 불필요. Token Scope Guide 섹션만 수정.

1.3 GitLab Token Scope Guide UI 상세

수정 후 예상 UI입니다. GitHub 버튼 클릭 시와 GitLab 버튼 클릭 시 콘텐츠가 전환됩니다.

🛡 API Token Scope Configuration Guide

When using a Personal Access Token (PAT) in GitLab, only the following permissions are needed.

GitHub GitLab
Minimum Required Scopes
api Full API access — required for uploading CI/CD results Required
read_api Read-only API access (alternative if write isn't needed) Minimum
read_repository Repository read access (for private repos) Optional
Recommended: Using the automatically provided CI_JOB_TOKEN within GitLab CI/CD pipelines is the safest approach — no separate PAT creation needed.

2.3 Getting Started

Getting Started Guide

필수 /docs/getting-started

목적: 신규 사용자가 회원가입부터 첫 테스트 케이스 실행까지 5분 내에 완료할 수 있도록 안내합니다.

문서 목차

섹션핵심 내용포함 요소
1. 회원가입 Email 가입 + Google SSO 방법. 이메일 인증 프로세스 스크린샷 2-3장, 단계별 설명
2. 첫 프로젝트 생성 Dashboard → "Create New Project" 클릭 → 프로젝트명, prefix 입력 → 생성 완료 스크린샷, prefix 규칙 설명 (예: "PRJ" → PRJ-1, PRJ-2)
3. 첫 Test Case 생성 프로젝트 → Test Cases 탭 → "New Test Case" 버튼 → 제목, 우선순위, Steps 입력 → 저장 스크린샷, 필드 설명 (title, priority, type, precondition, steps, expected result)
4. 첫 Test Run 실행 Test Runs 탭 → "Create Run" → TC 선택 → 실행 → Focus Mode에서 결과 입력 스크린샷, 상태값 설명 (passed/failed/blocked/retest/not_tested)
5. 다음 단계 팀원 초대, 마일스톤 설정, CI/CD 연동 링크 관련 문서 링크

2.4 API Reference

REST API Reference

필수 /docs/api/*

목적: Testably의 모든 REST API 엔드포인트를 문서화합니다. 각 엔드포인트에 대해 Method, Path, Parameters, Request/Response Body, 예제를 제공합니다.

공통 사항 (모든 API 페이지 상단에 포함)
Base URL: https://api.testably.app/v1 (또는 Supabase Edge Function URL)
Authentication: Authorization: Bearer YOUR_API_TOKEN
Content-Type: application/json
Rate Limit: 60 requests/min per token → 429 Too Many Requests
Error Format: { "error": "error_code", "message": "Human readable message" }

2.4.1 Authentication (/docs/api/authentication)

섹션내용
API Key 발급 방법Settings → API & Tokens → "New Token" → 이름 입력 → 토큰 생성 → 복사 및 안전 보관. 스크린샷 포함
Bearer Token 사용법curl -H "Authorization: Bearer testably_xxx..." 예제. 모든 요청에 필수
토큰 보안 가이드환경변수로 관리, 코드에 하드코딩 금지, CI/CD Secrets 활용 권장
토큰 무효화Settings → API & Tokens → Delete 버튼. 즉시 비활성화 (soft delete: is_active=false)

2.4.2 Projects API (/docs/api/projects)

GET /v1/projects 프로젝트 목록 조회
GET /v1/projects/:id 프로젝트 상세 조회
POST /v1/projects 프로젝트 생성
PATCH /v1/projects/:id 프로젝트 수정
DEL /v1/projects/:id 프로젝트 삭제

각 엔드포인트별 Request/Response 예제, 파라미터 설명, 에러 코드 포함

2.4.3 Test Cases API (/docs/api/test-cases)

GET /v1/projects/:pid/test-cases TC 목록 (필터: folder, priority, type, search)
GET /v1/projects/:pid/test-cases/:id TC 상세 조회
POST /v1/projects/:pid/test-cases TC 생성
PATCH /v1/projects/:pid/test-cases/:id TC 수정
DEL /v1/projects/:pid/test-cases/:id TC 삭제
POST /v1/projects/:pid/test-cases/bulk TC 벌크 생성

필드: title, folder_id, priority(critical/high/medium/low), type(manual/automated), precondition, steps[], expected_result, custom_id

2.4.4 Test Runs API (/docs/api/test-runs)

GET /v1/projects/:pid/runs Run 목록 조회
GET /v1/projects/:pid/runs/:id Run 상세 조회
POST /v1/projects/:pid/runs Run 생성 (TC IDs 포함)
PATCH /v1/projects/:pid/runs/:id Run 수정 (이름, 상태)
POST /v1/projects/:pid/runs/:id/close Run 종료

2.4.5 Test Results API (/docs/api/test-results)

GET /v1/runs/:rid/results Run의 결과 목록
PUT /v1/runs/:rid/results/:id 개별 결과 업데이트
POST /v1/runs/:rid/results/bulk 벌크 결과 업로드 (CI/CD용)

상태값: passed, failed, blocked, retest, not_tested. 각 결과에 note, elapsed(실행시간), author 포함 가능

2.4.6 Milestones API (/docs/api/milestones)

GET /v1/projects/:pid/milestones 마일스톤 목록
POST /v1/projects/:pid/milestones 마일스톤 생성
PATCH /v1/projects/:pid/milestones/:id 마일스톤 수정

2.4.7 Discovery Logs API (/docs/api/discovery-logs)

GET /v1/projects/:pid/sessions 세션 목록
POST /v1/projects/:pid/sessions 세션 생성
GET /v1/sessions/:sid/logs 로그 목록
POST /v1/sessions/:sid/logs 로그 추가

2.4.8 Members API (/docs/api/members)

GET /v1/projects/:pid/members 멤버 목록 (role 포함)
POST /v1/projects/:pid/members/invite 멤버 초대 (이메일)
PATCH /v1/projects/:pid/members/:uid 역할 변경
DEL /v1/projects/:pid/members/:uid 멤버 제거

역할: admin, member, viewer

2.5 CI/CD Integration

CI/CD Integration Guide

필수 /docs/cicd

목적: Settings → CI/CD Pipelines에 이미 있는 코드 스니펫을 바탕으로, 각 플랫폼별 상세 연동 가이드를 제공합니다.

섹션내용소스 (Settings 코드 기반)
1. 사전 준비 API Token 생성 → 환경변수 등록 (TESTABLY_URL, TESTABLY_TOKEN) Settings CI/CD 탭의 환경변수 등록 가이드 영역 기반
2. GitHub Actions 전체 YAML 코드 + 단계별 설명:
① Secrets에 TESTABLY_URL, TESTABLY_TOKEN 등록
② Workflow YAML 파일 생성 (.github/workflows/testably.yml)
③ curl로 결과 업로드 (POST 요청)
④ run_id, test_case_id, status 파라미터 설명 		getYAMLSnippet('github') 함수 기반
3. GitLab CI 전체 YAML 코드 + 단계별 설명:
① CI/CD Variables에 등록
② .gitlab-ci.yml 파일 작성
③ JUnit XML 아티팩트 연동
④ upload_results 스테이지 설명 		getYAMLSnippet('gitlab') 함수 기반
4. Python (pytest) 두 가지 방법:
Method 1: Function-based — report_result() 함수로 수동 업로드
Method 2: conftest.py — pytest hook 자동 수집 + 업로드
각 방법의 전체 코드 + 사용법 설명 		getPythonFunctionSnippet(), getPythonConftestSnippet() 기반
5. 공통 참고사항 상태값 목록: passed/failed/blocked/retest
run_id 확인 방법 (Testably UI에서 복사)
test_case_id 형식 (예: SUI-1)
업로드 API 엔드포인트 설명 		코드 스니펫 하단의 How to use 가이드 기반
Settings 코드에서 가져올 내용
Settings page.tsx의 getYAMLSnippet(), getPythonFunctionSnippet(), getPythonConftestSnippet() 함수에 이미 완성된 코드 스니펫이 있습니다. Documentation 페이지에서는 이 코드를 그대로 사용하되, 각 줄에 대한 상세 설명을 추가합니다.

2.6 Authentication

Authentication Guide

필수 /docs/api/authentication
섹션내용
1. API Key 발급 방법 경로: Settings → API & Tokens → "New Token"
단계:
① "New Token" 버튼 클릭
② Token 이름 입력 (예: "CI/CD Pipeline", "Local Development")
③ "Create" 클릭 → testably_xxxxxxxxxxxx 형태 토큰 생성
④ 토큰 복사 (클립보드 복사 버튼 사용)
⑤ 안전한 곳에 보관 (토큰은 다시 확인 불가하지 않지만, 보안상 환경변수 사용 권장)
스크린샷: New Token 모달 + 생성된 토큰 화면
2. Bearer Token 사용법 모든 API 요청의 헤더에 포함:
Authorization: Bearer testably_your_token_here

curl 예제:
curl -H "Authorization: Bearer testably_abc123..." \
     -H "Content-Type: application/json" \
     https://api.testably.app/v1/projects

JavaScript 예제:
fetch(url, { headers: { 'Authorization': `Bearer ${token}` } })
3. 보안 가이드 환경변수 사용 권장 (TESTABLY_TOKEN)
코드에 하드코딩 금지, .env 파일 사용 시 .gitignore에 추가
CI/CD에서는 GitHub Secrets / GitLab CI Variables 사용
토큰 유출 시 즉시 삭제 후 재발급
4. 에러 코드 401: Invalid or missing token
403: Token doesn't have access to this resource
429: Rate limit exceeded (60 req/min)

2.8 Webhooks

Webhook Integration Guide

중요 /docs/webhooks
섹션내용
1. Slack Webhook 설정 사전 준비: Slack Workspace에서 Incoming Webhook 생성 (api.slack.com/apps → Incoming Webhooks → Add New Webhook to Workspace)
Testably 설정:
① Settings → Integrations → "Add Integration"
② Type: Slack 선택
③ Project 선택
④ Webhook URL 입력 (https://hooks.slack.com/services/...)
⑤ Channel name (선택사항)
⑥ Events 선택
⑦ Save → Test 버튼으로 확인
2. Microsoft Teams Webhook 설정 사전 준비: Teams Channel → Connectors → "Incoming Webhook" 추가 → URL 복사
Testably 설정: Slack과 동일한 절차. Type을 "Microsoft Teams"로 선택
3. 지원 이벤트 7가지 이벤트 (현재 구현된 WEBHOOK_EVENTS 기반):
Team Activity: invitation_received, member_joined
Test Runs: run_created, run_completed
Milestones: milestone_started, milestone_completed, milestone_past_due
각 이벤트별 payload 예제 포함
4. 관리 기능 Toggle (활성/비활성), Edit, Delete, Test (테스트 메시지 전송), Logs (배달 로그 조회, 최근 50건)

2.9 Integrations (Jira)

Jira Integration Guide

중요 /docs/integrations
섹션내용
1. 사전 준비 Jira Cloud 계정 필요 (Server/Data Center는 추후 지원)
Jira API Token 생성: id.atlassian.com → Security → API tokens → Create token
필요 정보: Jira Domain, Email, API Token
2. Testably에서 설정 경로: Settings → Integrations → Jira Settings
티어: Starter 이상 필요 (Free는 잠금)
① Jira Domain 입력 (예: yourcompany.atlassian.net)
② Email 입력
③ API Token 입력
④ Issue Type 선택 (Bug, Task, Story 등)
⑤ "Test Connection" 으로 연결 확인
⑥ "Save Settings"
스크린샷: Jira 설정 폼 화면
3. 사용법 Test Case 상세 또는 Test Run 결과에서 "Create Jira Issue" 버튼 → 자동으로 Jira 이슈 생성
테스트 실패 시 빠르게 버그 리포트 가능
4. 문제 해결 연결 실패 시: Domain 확인, API Token 유효성, 네트워크
권한 오류: Jira 프로젝트 접근 권한 확인
Issue Type 없음: Jira 프로젝트 설정에서 Issue Type 확인
