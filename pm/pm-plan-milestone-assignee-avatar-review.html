2. 데이터 플로우 및 발견된 버그

2-1. Milestone 리스트 — Assignee 데이터 수집 방식

test_runs 전체 조회 → run.milestone_id로 그룹핑 → run.assignees[] 합집합 → milestoneRunAssignees Map → profiles 테이블 조회 (UUID) → AvatarStack 렌더

핵심: Milestone 자체에는 assigned_to 필드가 있지만 사실상 미사용(string | null, optional). 현재 아바타는 해당 Milestone에 연결된 모든 Test Run의 assignees를 합산한 결과를 표시. 즉, "이 Milestone의 담당자"가 아니라 "이 Milestone 하위 Run에 참여한 사람들"의 집합.

2-2. Milestone Detail — Run 카드 Assignee 데이터 수집 방식

run_testcase_assignees 테이블 → run_id별 assignee 수집 → fallback: test_cases.assignee → profiles 조회 (name/email) → AvatarStack 렌더

⚠️ 데이터 소스 불일치: Milestone 리스트는 test_runs.assignees[] (UUID 배열)를 사용하고, Milestone Detail은 run_testcase_assignees.assignee + test_cases.assignee (이름/이메일 문자열)를 사용. 동일 Run에 대해 서로 다른 사람이 표시될 수 있음.

2-3. 발견된 버그

🐛 Bug 1 — "?" 아바타 표시 문제

CEO가 보고한 ? 아바타 현상. Avatar.tsx의 getInitials() 함수에서 name과 email 모두 없으면 '?'를 반환. 원인: milestoneAssigneeProfiles Map에서 UUID로 프로필을 못 찾을 경우, name: undefined, email: undefined로 Avatar에 전달됨.

// Avatar.tsx line 29-39 export function getInitials(name?: string, email?: string): string { if (name && name.trim()) { ... return initials; } if (email) return email.slice(0, 2).toUpperCase(); return '?'; // ← 이 케이스가 발생 }

🐛 Bug 2 — Milestone Detail의 profile 조회 로직 오류

Milestone Detail에서 run_testcase_assignees.assignee는 이름/이메일 문자열인데, 이를 profiles 테이블에서 full_name.eq.${n},email.eq.${n}로 OR 조회. 만약 assignee 값이 UUID라면 full_name이나 email과 매칭되지 않아 프로필이 누락되고, 이 역시 ? 표시 원인이 됨.

🐛 Bug 3 — Sub-Milestone에 직접 연결된 Run이 없으면 아바타 비어 있음

Progress는 parent에서 fallback 상속하지만, assignee는 상속 로직 없음. Sub-Milestone에 직접 milestone_id가 걸린 Run이 없으면 아바타가 표시되지 않아 Parent와 Sub 간 정보 비대칭 발생.

3. UX 타당성 평가

3-1. "Milestone 카드에 assignee 아바타를 표시하는 것이 의미가 있는가?"

평가 기준현재 상태평가
정보의 정확성 Milestone 담당자가 아닌 "Run 참여자 합산"을 표시 ❌ Misleading
시각적 가독성 xs(20px) 사이즈, 겹침 적용시 12px 실효 영역 ❌ 인지 어려움

3-2. 핵심 문제 요약

근본적 문제: 현재 아바타가 보여주는 정보는 "Milestone의 담당자(Owner)"가 아니라 "하위 Run들의 assignees 합집합"이며, 이 차이를 사용자가 알 수 없음. 카드에 "Assigned to" 같은 라벨도 없어서 아바타가 무엇을 의미하는지 모호함.

4. 방향 A — Assignee 아바타 제거

Milestone 리스트 카드에서 AvatarStack을 완전 제거. 카드는 name, date, status, progress만 표시하여 정보 밀도를 낮추고 핵심에 집중.

4-1. 변경 범위

파일변경 사항LOC 예상

② fetchData()에서 assignee 수집/프로필 조회 로직 삭제 (line 246-267)
③ renderMilestone 하단 AvatarStack 삭제 (line 637-650)
④ renderSubMilestone AvatarStack 삭제 (line 464-477)
⑤ Avatar, AvatarStack import 제거 (line 10) 		~40줄 삭제

✅ 장점

  • Misleading 정보 제거 — "이 사람이 Milestone 담당자"라는 오해 방지
  • 카드 정보 밀도 감소 — 핵심 정보(이름, 날짜, 상태, 진행률)에 집중
  • "?" 아바타 버그 근본 해결
  • 네트워크 요청 1회 절감 (profiles 조회 불필요)
  • 코드 40줄 삭제로 유지보수 부담 감소

❌ 단점

  • Milestone 리스트에서 "누가 참여하고 있나" 정보를 전혀 볼 수 없음
  • Detail 페이지까지 들어가야 참여자 확인 가능
  • 팀 관리자 관점에서 한눈에 부하 분산 파악 불가
  • 카드 하단 우측이 비어서 레이아웃 밸런스 변화

7. Milestone Detail — Run 카드에도 동일 기준 적용 필요한지?

7-1. 현재 구현

Milestone Detail 페이지의 Run 카드에는 AvatarStack size="xs" max={4}로 각 Run의 assignees가 표시됨. 데이터 소스는 run_testcase_assignees 테이블 + test_cases.assignee fallback.

7-2. Run 카드와 Milestone 카드의 본질적 차이

속성Milestone 카드Run 카드 (Detail 내부)
Assignee의 의미 하위 Run들의 참여자 합산 (간접/파생) 해당 Run에 직접 배정된 테스터 (직접)
정보 정확성 모호함 — "누구 담당?" vs "누가 참여?" 명확함 — "이 Run을 실행하는 사람"
사용자 니즈 리스트 스캔 시 담당자 확인은 부차적 Run 실행 시 배정자 확인은 핵심적
경쟁 제품 참고 TestRail: Milestone 리스트에 아바타 없음 TestRail: Run에 "Assigned To" 표시함

결론: Milestone Detail의 Run 카드 아바타는 유지하되 개선해야 함. Run 카드의 assignee는 직접적이고 명확한 정보이므로 Milestone 리스트 카드와 동일 기준을 적용할 필요 없음. 다만 아래 개선은 공통 적용 권장:

7-3. Run 카드 공통 개선 사항

#항목내용
R-1 "?" 버그 수정 프로필 조회 로직을 UUID 기반으로 통일 (현재 name/email OR 쿼리 → UUID .in() 쿼리)
R-2 사이즈 sm 통일 project-runs 페이지는 이미 Avatar size="sm" 사용. milestone-detail도 xs→sm
R-3 "Assigned to" 라벨 project-runs처럼 아바타 앞에 "Assigned to" 텍스트 추가

8. 권장안 및 구현 로드맵

🎯 PM 권장: 방향 A (제거) + Run 카드 개선

8-1. 권장 이유

Milestone 리스트 카드의 아바타는 파생 데이터(Run assignees 합산)를 마치 Milestone 담당자인 것처럼 보여주고 있어 UX적으로 misleading합니다. xs(20px) 사이즈로 가독성도 떨어지고, "?" 버그까지 발생 중입니다. 방향 B로 개선하려면 ~2.75일의 공수가 드는 반면, 근본적으로 "파생 데이터"라는 한계는 해소되지 않습니다.

반면 Milestone Detail 내부의 Run 카드 아바타는 "이 Run의 직접 배정자"로 의미가 명확하므로 유지 + 개선이 타당합니다. 이렇게 하면 정보 계층이 자연스럽게 형성됩니다:

Milestone 리스트 (무엇이 진행 중인가? → 이름, 상태, 진행률) → Milestone Detail (누가 무엇을 하고 있나? → Run별 assignees) → Run Detail (각 테스트 결과는?)

8-3. Phase 3 상세 — Milestone Owner 개념 (향후 발전 방향)

만약 Milestone 카드에 "사람"을 보여주고 싶다면, 현재처럼 Run assignees를 합산하는 것이 아니라 Milestone 자체에 Owner(책임자) 개념을 도입하는 것이 올바른 접근입니다. 이렇게 하면 "이 Milestone은 누가 책임지고 있는가?"라는 질문에 정확하게 답할 수 있습니다.

-- Phase 3: DB 마이그레이션 ALTER TABLE milestones ADD COLUMN owner_id UUID REFERENCES profiles(id); -- Milestone 카드에 단일 Avatar 표시 <Avatar userId={milestone.owner_id} name={ownerProfile?.full_name} email={ownerProfile?.email} photoUrl={ownerProfile?.avatar_url} size="sm" title={`Owner: ${ownerProfile?.full_name}`} />
