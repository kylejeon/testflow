1
Add Result 모달에서도 Failed 시 Auto-create 동작
Bug Fix
현재 문제: Auto-create Jira Issue는 Quick Status 변경(handleStatusChange)과 Focus Mode에서만 트리거됩니다. Add Result 모달(handleSubmitResult)에서 Failed를 선택하고 제출해도 자동 생성이 작동하지 않습니다.

현재 동작 분석

📄 src/pages/run-detail/page.tsx

함수 위치 Auto-create 포함 설명
handleStatusChange line 587–645 ✅ Yes Quick Status 드롭다운 변경 시 호출
handleFocusStatusChange line 1472–1490 ✅ Yes 내부적으로 handleStatusChange 호출
handleSubmitResult line 877–970 ❌ No Add Result 모달 제출 — Auto-create 누락
해결 방안: handleSubmitResult에서 test_results INSERT 성공 후, status가 'failed'일 때 동일한 auto-create 로직을 실행합니다.

구현 위치

handleSubmitResult 함수 내부, line 927 이후 (test_results INSERT 성공 직후, issues 업데이트 로직 전).

이때 insert된 result의 data.id와 기존 issues 배열을 활용하여 auto-create 조건을 검사합니다.

Auto-create 조건 (handleStatusChange와 동일)

  1. resultStatus === 'failed'
  2. jiraSettings.auto_create_on_failure !== 'disabled'
  3. jiraSettings.project_key 존재
  4. all_failures 모드: 항상 생성
  5. first_failure_only 모드: 기존 issues가 없을 때만 생성

참조할 기존 코드

// handleStatusChange 내 auto-create 블록 (line 612–645) if (newStatus === 'failed' && jiraSettings?.auto_create_on_failure && jiraSettings.auto_create_on_failure !== 'disabled' && jiraSettings.project_key) { const tc = testCases.find(t => t.id === testCaseId); if (tc) { const existingIssues = newResultData?.issues || []; const shouldCreate = jiraSettings.auto_create_on_failure === 'all_failures' || (jiraSettings.auto_create_on_failure === 'first_failure_only' && existingIssues.length === 0); if (shouldCreate) { // ... create-jira-issue 호출 } } }
2
Auto-create 성공/실패 시 Toast 알림
New Feature
현재 문제: Auto-create가 성공하면 아무 피드백 없이 issues 배열에 key가 추가되기만 합니다. 실패하면 console.warn만 찍히고 사용자에게는 아무 표시도 없습니다.
해결 방안: 간단한 인라인 Toast 컴포넌트를 run-detail 내부에 구현하고, auto-create 결과에 따라 성공/실패 토스트를 표시합니다.

Toast 디자인 스펙

✅ 성공 Toast

  • 배경: #ECFDF5 (green-50)
  • 아이콘: ri-check-line (green)
  • 텍스트: "Jira issue {ISSUE-KEY} created automatically"
  • 3초 후 자동 사라짐

❌ 실패 Toast

  • 배경: #FEF2F2 (red-50)
  • 아이콘: ri-error-warning-line (red)
  • 텍스트: "Failed to auto-create Jira issue"
  • 5초 후 자동 사라짐 (실패는 더 오래 표시)

구현 방식

외부 라이브러리 없이 run-detail 내부에 state 기반 Toast를 구현합니다:

// State 추가 const [toast, setToast] = useState<{ type: 'success' | 'error'; message: string; } | null>(null); // Toast 표시 함수 const showToast = (type: 'success' | 'error', message: string) => { setToast({ type, message }); setTimeout(() => setToast(null), type === 'success' ? 3000 : 5000); }; // auto-create 성공 시: showToast('success', `Jira issue ${jiraData.issue.key} created automatically`); // auto-create 실패 시: showToast('error', 'Failed to auto-create Jira issue');

Toast UI 위치

return JSX 최상단, fixed 포지션으로 화면 상단 중앙에 표시합니다.

{/* Toast Notification */} {toast && ( <div className="fixed top-4 left-1/2 -translate-x-1/2 z-50 ..."> <i className={toast.type === 'success' ? 'ri-check-line' : 'ri-error-warning-line'} /> <span>{toast.message}</span> </div> )}

  2. handleSubmitResult (신규 auto-create — 지시 #1)

Settings UI: Summary/Description 템플릿 커스터마이즈
New Feature
현재 문제: Auto-create 시 Summary는 하드코딩([Auto] Test Failed: {tc.title})이고, Description은 buildAutoJiraDescription이 고정 포맷입니다. 사용자가 커스터마이즈할 수 없습니다.
해결 방안: Settings의 Jira 탭에 Summary/Description 템플릿 입력 UI를 추가하고, 변수 삽입 시스템을 구현합니다.

C. Settings UI 디자인

위치: 📄 src/pages/settings/page.tsx — Auto-create 드롭다운 (line 1850) 아래에 추가

UI 구성:
  1. Summary Template — 단일 text input, 위에 설명 텍스트 + 변수 태그 버튼들
  2. Description Template — textarea (6줄), 위에 설명 텍스트 + 변수 태그 버튼들
  3. 각 변수 태그를 클릭하면 현재 커서 위치에 변수가 삽입됨
  4. "Reset to Default" 버튼으로 기본 템플릿 복원

조건부 표시: autoCreateOnFailure !== 'disabled'일 때만 템플릿 UI를 보여줍니다.

4
buildAutoJiraDescription에 Expected Result 포함
Bug Fix
현재 문제: buildAutoJiraDescription (line 572–579)이 Steps와 Precondition만 포함하고 expected_result를 빠뜨리고 있습니다. TestCase 인터페이스에는 필드가 정의되어 있음.

현재 코드

const buildAutoJiraDescription = (tc: TestCase): string => { const steps = tc.steps || 'No steps defined'; return `*Auto-created by Testably*\n\n` + `Test Case: ${tc.title}\n` + `Run: ${run?.name || 'Unknown'}\n` + `Priority: ${tc.priority || 'Medium'}\n\n` + `--- Steps ---\n${steps}\n\n` + (tc.precondition ? `--- Precondition ---\n${tc.precondition}` : ''); };
해결 방안: Expected Result 섹션을 Steps와 Precondition 사이에 추가합니다.

수정 후 코드

const buildAutoJiraDescription = (tc: TestCase): string => { const steps = tc.steps || 'No steps defined'; const expectedResult = tc.expected_result || 'Not specified'; return `*Auto-created by Testably*\n\n` + `Test Case: ${tc.title}\n` + `Run: ${run?.name || 'Unknown'}\n` + `Priority: ${tc.priority || 'Medium'}\n\n` + `--- Steps ---\n${steps}\n\n` + `--- Expected Result ---\n${expectedResult}\n\n` + (tc.precondition ? `--- Precondition ---\n${tc.precondition}` : ''); };

참고: 지시 #3의 템플릿 시스템이 구현되면 이 함수는 fallback 용도로만 사용됩니다. 그러나 템플릿이 비어있거나 DB 마이그레이션 전 호환을 위해 이 수정은 반드시 필요합니다.

테스트 시나리오

1. Add Result 모달 Auto-create (#1)

  • Settings에서 auto_create_on_failure = 'all_failures' 설정
  • Run detail에서 TC의 "Add Result" 클릭 → Status=Failed 선택 → Submit
  • Jira issue가 자동 생성되고, issues 배열에 key가 추가되는지 확인
  • Toast가 표시되는지 확인 (#2)
  • first_failure_only 모드에서 이미 issue가 있는 TC에 대해 생성되지 않는지 확인

2. Toast 알림 (#2)

  • Auto-create 성공 시 초록색 토스트가 3초간 표시
  • 잘못된 Jira credentials 사용 시 빨간색 토스트가 5초간 표시
  • Quick Status 변경과 Add Result 모달 모두에서 토스트 동작 확인

3. 템플릿 시스템 (#3)

  • Settings에서 Summary/Description 템플릿 수정 → Save
  • 변수 태그 클릭 시 커서 위치에 삽입
  • Reset to Default 동작 확인
  • Auto-create 시 커스텀 템플릿이 적용된 Jira issue 확인
  • 템플릿이 비어있을 때 fallback 동작 확인

4. Expected Result (#4)

  • expected_result가 있는 TC에서 auto-create → Description에 포함되는지 확인
  • expected_result가 없는 TC → "Not specified"로 표시되는지 확인
