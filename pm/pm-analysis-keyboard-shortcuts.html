1 Executive Summary

Testably 코드베이스에서 50+ 개 키보드 액션을 16개 파일에서 확인했습니다. 이 중 직접적인 브라우저 충돌은 0건이지만, 잠재적 충돌 및 접근성 위험 3건을 식별했습니다.

✅ 핵심 결론: Testably의 현재 단축키 설계는 전반적으로 양호합니다. 대부분의 단일키 단축키는 input/textarea에서 비활성화되고, 수정자 키(Ctrl/Cmd) 조합은 브라우저와 겹치지 않습니다. 다만 N, R 단일키 글로벌 단축키는 접근성 측면에서 개선이 필요합니다.

단축키 가이드 팝업: ? 키 트리거 + 헤더 버튼 이중 접근 방식을 권장합니다. GitHub, Gmail, Linear 등 모든 주요 SaaS가 이 패턴을 사용하며, Testably에는 이미 useKeyboardShortcuts.ts에 ? → Show Shortcuts Help 핸들러가 구현되어 있습니다.

3-2. 충돌 상세 분석

⚠️ 잠재적 충돌 #1: Ctrl/Cmd+K — Command Palette
Chrome에서 Cmd+K는 주소창 포커스입니다. Testably는 preventDefault()를 호출하여 브라우저 기본 동작을 차단합니다.
위험도: 낮음 — 이 패턴은 Notion, Linear, VS Code, Slack 등 모든 주요 SaaS가 동일하게 사용합니다. 사실상 업계 표준입니다.
ℹ️ 잠재적 이슈 #2: N / R 단일키 — 대시보드
브라우저와 직접 충돌하지는 않지만, 스크린 리더나 보조 기술 사용 시 단일키 단축키가 탐색을 방해할 수 있습니다.
위험도: 중간 — GitHub처럼 "접근성 설정에서 단일키 단축키 비활성화" 옵션 추가를 권장합니다.
ℹ️ 잠재적 이슈 #3: Focus Mode 단일키 밀도
PFBRSNJKCH — 10개 단일키가 Focus Mode에서 활성화됩니다.
위험도: 낮음 — Focus Mode는 전용 전체화면이므로 일반 입력과 분리됩니다. Input 내 차단 로직도 정상 작동합니다. 하지만 사용자가 처음 접할 때 학습 부담이 있으므로 단축키 가이드 팝업이 필수적입니다.

8-2. UX 가치 분석

  • 발견성(Discoverability): 현재 Testably의 단축키는 Focus Mode UI의 kbd 힌트를 제외하면 전혀 노출되지 않습니다. 가이드 팝업이 없으면 사용자가 단축키 존재 자체를 모를 가능성이 높습니다.
  • 학습 곡선 완화: Focus Mode에 10개 단일키가 있어 학습 부담이 큽니다. 가이드 팝업은 이 부담을 크게 줄여줍니다.
  • 경쟁 우위: TestRail, Qase, Zephyr 중 인앱 단축키 가이드를 제공하는 제품이 없습니다. Testably가 QA 도구 최초로 이를 구현하면 "파워 유저 친화적" 이미지를 강화할 수 있습니다.
  • 온보딩 보조: 신규 사용자가 첫 Focus Mode 진입 시 자동으로 가이드 팝업을 보여주는 옵션도 검토 가능합니다 (Phase 2).

8-3. 충돌 해결 요약

✅ 결론: 현재 단축키 변경 불필요
직접적인 브라우저 충돌 0건. Cmd+K는 업계 표준 패턴. 단일키(N/R)는 접근성 토글로 Phase 2에서 대응. Focus Mode 단축키는 격리된 환경이므로 안전.

9 CEO Q&A

Q: 브라우저 단축키와 충돌하는 것이 있나?

A: 직접 충돌 0건. Cmd+K는 브라우저 주소창과 겹치지만, Notion/Linear/Slack/VS Code 모두 동일하게 override하는 업계 표준 패턴입니다. preventDefault()로 정상 차단 중.

Q: Focus Mode 단축키(P/F/B/R 등)가 일반 타이핑과 겹치지 않나?

A: INPUT/TEXTAREA에서는 모두 차단됩니다 (Line 275-286). 노트 입력 중에는 Escape과 Cmd+Enter만 동작합니다.

Q: 단축키 가이드 팝업은 바로 만들 수 있나?

A: 네. ? 키 핸들러가 이미 구현되어 있어(useKeyboardShortcuts.ts Line 61), 모달 UI만 만들면 됩니다. shortcuts.ts 중앙 배열 포함 약 5시간 작업량입니다.

Q: 경쟁사 QA 도구는 이런 기능이 있나?

A: TestRail, Qase, Zephyr 모두 인앱 단축키 가이드가 없습니다. 외부 도움말 센터에서만 확인 가능합니다. Testably가 QA 도구 최초로 GitHub/Linear 수준의 인앱 가이드를 제공하면 차별화 포인트가 됩니다.

Q: 향후 Run Workflow 단축키 추가 시 충돌 위험은?

A: A (Add Result), 1~5 (Step 결과), Ctrl+Enter (모달 제출) — 모두 브라우저와 충돌 없습니다. shortcuts.ts에 미리 예약 등록해두면 관리가 수월합니다.
